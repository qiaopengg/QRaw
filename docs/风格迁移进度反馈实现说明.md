# 风格迁移进度反馈实现说明

## 概述

本文档说明了 RapidRAW 风格迁移功能的实时进度反馈系统实现。该系统解决了用户在执行风格迁移时无法判断进程是否卡死还是正在计算的问题。

## 问题背景

**用户痛点：**

- 风格迁移处理时间较长（大图可能需要数分钟）
- 用户无法判断进程是否正常运行还是已经卡死
- 缺乏进度反馈导致用户体验差

## 解决方案架构

### 1. 三层进度反馈系统

```
┌─────────────────────────────────────────────────────────────┐
│                      前端 (ChatPanel.tsx)                    │
│  - ThinkingBlock 组件显示实时进度                             │
│  - 监听 'style-transfer-stream' 事件                         │
│  - 自动滚动到最新消息                                         │
└─────────────────────────────────────────────────────────────┘
                              ↑
                              │ Tauri Event
                              │
┌─────────────────────────────────────────────────────────────┐
│              Rust 后端 (style_transfer_runtime.rs)           │
│  - 监听 Python 服务的 stdout                                 │
│  - 捕获 [PROGRESS] 前缀的日志                                │
│  - 通过 emit() 转发到前端                                    │
└─────────────────────────────────────────────────────────────┘
                              ↑
                              │ stdout
                              │
┌─────────────────────────────────────────────────────────────┐
│           Python 服务 (app_fixed.py)                         │
│  - 在关键步骤输出 [PROGRESS] 日志                            │
│  - 包含百分比和当前操作描述                                   │
└─────────────────────────────────────────────────────────────┘
```

### 2. 进度阶段划分

#### 单图处理流程 (`_run_single`)

| 进度 | 阶段         | 说明                         |
| ---- | ------------ | ---------------------------- |
| 0%   | 开始         | 接收请求，验证参数           |
| 5%   | 加载图像     | 读取参考图和内容图           |
| 10%  | 准备控制图像 | 生成 Canny 边缘检测图        |
| 20%  | SDXL Base    | 运行基础模型推理             |
| 70%  | SDXL Refiner | 运行精炼模型（如果启用）     |
| 80%  | 推理完成     | 单阶段推理完成（无 Refiner） |
| 90%  | Refiner 完成 | 精炼模型完成                 |
| 95%  | 后处理       | 值域归一化                   |

#### 分块处理流程 (`_run_tiled`)

| 进度    | 阶段     | 说明                        |
| ------- | -------- | --------------------------- |
| 10%-90% | 分块处理 | 每个 tile 占用 80% 进度空间 |
| 90%     | 融合块   | 将所有 tile 融合为完整图像  |

**示例：** 24 个 tile 的处理

- Tile 1/24: 10% + (0/24) \* 80% = 10%
- Tile 12/24: 10% + (11/24) \* 80% = 47%
- Tile 24/24: 10% + (23/24) \* 80% = 87%

#### 后处理流程 (`style_transfer`)

| 进度 | 阶段     | 说明                                 |
| ---- | -------- | ------------------------------------ |
| 92%  | 色彩对齐 | Luminance-aware Mapping + Tone Curve |
| 94%  | RAW 融合 | 保持动态范围和细节                   |
| 96%  | 保存文件 | 输出 16-bit TIFF 和预览图            |
| 100% | 完成     | 风格迁移完成                         |

### 2. Rust 后端转发（已实现）

Rust 后端通过 `emit_style_transfer_status()` 函数将进度转发到前端：

```rust
fn emit_style_transfer_status(app_handle: &tauri::AppHandle, text: impl Into<String>) {
    let _ = app_handle.emit(
        "style-transfer-stream",
        json!({
            "chunk_type": "thinking",
            "text": text.into(),
            "result": Value::Null
        }),
    );
}
```

**工作原理：**

1. Python 服务输出 `[PROGRESS]` 日志到 stdout
2. Rust 后端捕获 stdout 输出
3. 通过 Tauri Event 系统发送到前端
4. 前端监听 `style-transfer-stream` 事件

## 性能影响

### 进度输出开销

- **stdout 输出：** 每次 `print()` 约 0.1-0.5ms
- **总开销：** 对于 24 个 tile 的处理，约增加 10-20ms
- **相对影响：** < 0.1%（总处理时间通常为数分钟）

### 优化建议

1. **避免高频输出：** 不要在循环内部每次迭代都输出进度
2. **批量更新：** 每处理 N 个 tile 输出一次进度
3. **异步输出：** 考虑使用异步日志系统（如果性能成为瓶颈）

## 故障排查

### 问题 1: 进度不更新

**症状：** 前端 ThinkingBlock 不显示进度信息

**排查步骤：**

1. 检查 Python 服务是否正常输出 `[PROGRESS]` 日志

   ```bash
   # 查看 Python 服务日志
   tail -f /path/to/python/service.log
   ```

2. 检查 Rust 后端是否捕获 stdout

   ```rust
   // 确认 emit_style_transfer_status 被调用
   println!("Emitting progress: {}", text);
   ```

3. 检查前端是否监听事件
   ```typescript
   // 添加调试日志
   console.log('Received progress:', chunkText);
   ```

### 问题 2: 进度卡在某个百分比

**症状：** 进度显示到某个百分比后不再更新

**可能原因：**

1. Python 服务崩溃或挂起
2. 某个 tile 处理失败
3. 内存不足导致进程被杀

**排查步骤：**

1. 检查 Python 服务进程状态

   ```bash
   ps aux | grep python
   ```

2. 查看系统资源使用

   ```bash
   top -p <python_pid>
   ```

3. 检查错误日志
   ```bash
   grep ERROR /path/to/service.log
   ```

### 问题 3: 进度百分比不准确

**症状：** 进度跳跃或倒退

**可能原因：**

1. 分块数量计算错误
2. 进度计算公式错误

**修复方法：**

```python
# 确保进度单调递增
last_progress = 0
def emit_progress(pct, msg):
    global last_progress
    pct = max(pct, last_progress)
    print(f"[PROGRESS] {pct}% - {msg}...")
    last_progress = pct
```
