# RapidRAW 风格迁移技术架构 V3.1（真实场景重构版）

---

# 一、文档目的

本文档用于替换此前过于理想化的“最终工业落地版”描述。

新的目标不是继续追求抽象意义上的“高质量最终方案”，而是先回答下面三个真实问题：

```text
1. 用户能否在可接受时间内得到结果
2. 结果是否稳定，不会死黑、灰度化、色彩丢失
3. 功能是否符合专业摄影工作流，而不是演示型 AI 功能
```

因此，本文档从“理想方案说明”调整为“失败复盘后的可执行重构方案”。

---

# 二、当前结论

## 2.1 当前版本定性

当前生成式风格迁移链路可定性为：

- 产品结果失败
- 当前实现不可上线
- 技术方向部分可保留
- 工程落地方式需要重构

## 2.2 失败原因

当前方案在真实使用场景中暴露出以下根本问题：

- 单次等待时间可达 20 到 30 分钟，超出专业修图场景可接受范围
- 结果稳定出现死黑、色彩丢失、灰度化，说明输出链路存在严重质量缺陷
- 生成式输出与 RAW 非破坏性编辑工作流边界不清晰
- 文档承诺与实际代码执行路径不一致，导致产品语义失真

## 2.3 新目标

从本版本开始，风格迁移功能的产品目标调整为：

```text
先可用，再高质量
先稳定，再扩展
先预览，再高分辨率导出
```

# 三、产品边界重新定义

## 3.1 风格迁移分为两类能力

### A. 分析式风格迁移

定义：

- 基于参考图分析风格特征
- 输出调色参数建议
- 不生成新像素内容
- 继续工作在 RAW/调色工作流内

适用场景：

- 快速风格匹配
- 可控修图
- 专业摄影工作流

### B. 生成式风格迁移

定义：

- 基于参考图和当前图生成新的 RGB 结果图
- 输出是衍生图像，不是 RAW 本体
- 结果属于“生成式重绘/风格化结果”

适用场景：

- 创意风格预览
- 艺术化变体生成
- 非严格 RAW 还原场景

## 3.2 重要边界

必须明确：

```text
生成式风格迁移输出的是新的 16-bit RGB 衍生图
不是 RAW 本体，也不是 RAW 非破坏性编辑结果
```

这条边界必须在产品、文档、UI、导出命名中全部保持一致。

# 四、重构后的核心原则

新的风格迁移系统必须满足以下硬约束。

## 4.1 时间约束

- 预览结果目标控制在 10 到 30 秒内
- 30 到 120 秒仅属于生成式预览或高质量导出的可接受区间，不属于默认体验目标
- 超过 2 分钟即判定为默认交互路径不可接受

### 4.1.1 SLA 分层

| 路径           | 目标耗时     | 可接受耗时   | 超时策略                                          |
| -------------- | ------------ | ------------ | ------------------------------------------------- |
| 分析式风格迁移 | 1 到 5 秒    | 5 到 10 秒   | 超过 10 秒视为异常，需要排查本地分析链路          |
| 生成式风格预览 | 10 到 30 秒  | 30 到 120 秒 | 超过 2 分钟默认失败，或必须由用户明确确认继续等待 |
| 高质量导出     | 30 到 120 秒 | 2 分钟以上   | 超过 2 分钟不得静默继续，必须由用户明确确认       |

## 4.2 质量约束

- 不允许出现死黑图
- 不允许出现灰度化结果
- 不允许无提示地输出明显异常图像
- 不允许在 tile 失败后继续拼接假结果

## 4.3 产品约束

- 默认路径必须是“先预览后确认”
- 不允许默认对全分辨率原图直接执行慢速生成式推理
- 不允许在核心依赖缺失时静默降级并输出伪成功结果

补充说明：

- 对生成式路径，“先预览后确认”指先生成预览结果，再决定是否应用为衍生图或进入高质量导出
- 对分析式路径，“先预览后确认”等价于先得到参数建议，再决定是否应用到当前 RAW 工作流

## 4.4 工程约束

- 文档声明的能力必须与实际执行路径一致
- 健康检查返回的能力必须是真实可用能力
- 任何 fallback 都必须在 UI 中明确告知用户

## 4.5 集成约束

由于本项目的基础编辑能力来自上游开源项目，所有新增 AI 能力都必须遵循统一的无侵入集成原则。

这里的 AI 能力至少包括：

- AI 修图
- AI 选图
- 风格迁移

这些能力必须采用同一套开发模式，而不能各自选择不同的接入方式。

统一原则如下：

- 不直接侵入上游主工作流核心实现
- 不修改现有 RAW 编辑、预览、导出主链路的既有语义
- 新增能力必须拆分为可独立启停的模块
- 与现有项目集成时优先采用模块引用和接口适配，而不是大面积改写基建代码
- 当任意 AI 模块被移除时，原有编辑器、预览、导出、sidecar、preset 功能必须仍可独立正常工作
- 现有上游基建代码必须保持可持续同步更新，避免因本地深度侵入而造成长期冲突
- 新增 AI 功能必须优先考虑“可拔出、可替换、可单独迭代”，而不是“快速嵌入后再修补”

必须明确：

```text
AI 修图、AI 选图、风格迁移都只是挂载在现有基建之上的可拔插子系统
不是重写现有基建的替代系统
```

## 4.6 AI 子系统统一开发模式

从本版本开始，项目中所有新增 AI 功能一律遵循以下开发模式：

- 基建归基建，AI 归 AI
- 上游可同步，功能可迭代
- 默认模块化，拒绝隐式耦合
- 先接口适配，再能力扩展

工程上必须满足：

- AI 功能以独立模块目录存在
- AI 功能通过显式接口接入现有主画布、预览、导出、图库、sidecar、preset 系统
- AI 功能内部状态不得直接污染现有基建的全局状态模型
- 同步上游开源项目时，不应因 AI 功能改动而大面积冲突
- 任意 AI 模块应支持单独关闭、单独替换、单独演进
- 我们自己集成的新增用户可见功能必须接入项目现有多语言体系，不允许写死单语言文案
- 我们自己集成的前端新增代码必须严格使用 TypeScript，不允许以 JavaScript 形式绕过类型系统
- 我们自己集成的功能必须严格遵守代码质量规范，不允许因快速集成 AI 功能而降低可维护性

这条原则不是仅对风格迁移生效，而是对整个 AI 子系统长期生效。

# 五、保留与删除

## 5.1 保留的技术方向

- Rust + React 作为产品主工作流
- Python 服务承载独立 AI 推理
- 分析式与生成式双轨能力并存
- IP-Adapter 作为风格参考主方案
- ControlNet 作为结构约束组件

## 5.2 明确删除的错误假设

- 删除“允许 30 到 120 秒推理作为默认产品体验”的错误假设
- 删除“当前版本已具备工业级最终落地能力”的结论
- 删除“默认对大图直接做分块扩散”的交互设计
- 删除“核心风格注入失败后继续输出结果”的容错策略
- 删除“tile 失败以黑块补偿”的实现思路

# 六、重构后的系统架构

## 6.1 架构核心思想

```text
Rust = 工作流与体验控制中心
Python = 可中断、可降级、可验证的推理服务
```

## 6.2 系统结构

```
┌──────────────────────────────────┐
│ RapidRAW (Rust + React)          │
│ - RAW 编辑                       │
│ - 参考图选择                     │
│ - 模式切换                       │
│ - 任务状态与错误反馈             │
└────────────────┬─────────────────┘
                 │ IPC / HTTP / SSE
┌────────────────▼─────────────────┐
│ Python Inference Service         │
│ - Preview Pipeline               │
│ - Validation Layer               │
│ - Optional Export Pipeline       │
└────────────────┬─────────────────┘
                 │
┌────────────────▼─────────────────┐
│ GPU Runtime                      │
│ - FP16                           │
│ - SDXL + IP-Adapter + ControlNet │
└──────────────────────────────────┘
```

## 6.3 模块职责

### Rust 侧职责

- 管理模式和工作流
- 决定是走分析式还是生成式
- 统一承接任务状态、错误、超时、取消
- 控制输出文件管理和导入逻辑

### Python 侧职责

- 只负责模型推理与结果验证
- 不决定产品语义
- 不在关键失败时伪造可用输出
- 向上层报告明确状态：成功、失败、依赖缺失、预热中

## 6.4 模块化集成方式

为避免侵入上游基建，风格迁移相关能力必须按独立模块拆分，并通过引用式集成接入现有系统。

推荐拆分如下：

- `style-transfer-analysis-adapter`
  - 负责参考图分析、参数建议、preset 生成
  - 输出为当前项目可直接消费的调整参数或 sidecar 数据
- `style-transfer-preview-adapter`
  - 负责生成式预览前置流程、请求组装、预览结果接收、结果校验
  - 不直接改写现有编辑器预览主链路
- `style-transfer-export-adapter`
  - 负责高质量导出阶段与现有导出系统对接
  - 输出必须通过现有导出管线进行保存和归档
- `style-transfer-asset-registry`
  - 负责生成式预览结果、衍生图、正式导出结果的身份与归属管理
  - 不直接替代现有图库或 sidecar 模型
- `style-transfer-quality-guard`
  - 负责亮度、色彩、黑图、灰片、tile 失败等质量校验
  - 作为独立守门模块接入预览和导出两条链路
- `style-transfer-task-orchestrator`
  - 负责长任务状态、取消、超时、重试、结果状态流转
  - 不与现有导出任务模型硬耦合

模块接入方式建议如下：

```text
现有 RapidRAW 基建
  ├─ 预览能力
  ├─ 导出能力
  ├─ sidecar / virtual copy
  ├─ preset 系统
  └─ 主画布 UI
          ↑
          │ 接口适配 / 模块引用
          │
风格迁移子系统
  ├─ analysis adapter
  ├─ preview adapter
  ├─ export adapter
  ├─ asset registry
  ├─ quality guard
  └─ task orchestrator
```

模块化设计的核心要求：

- 子模块只通过明确接口读写现有基建
- 子模块不得直接改写现有缓存、导出、图库、RAW 渲染的核心语义
- 子模块故障时必须可整体禁用，而不影响原有基建继续运行

# 七、重构后的功能路线

## 7.1 路径选择原则

风格迁移不应默认替用户决定“先走分析式”还是“先走生成式”。

系统必须让用户显式选择路径，并在选择前明确展示：

- 当前路径类型：分析式或生成式
- 预计等待时间区间
- 结果类型：参数建议、临时预览、衍生图、正式导出
- 是否需要额外导出时间

推荐交互规则：

- 默认展示两种路径，而不是默认直接执行其中一种
- 分析式路径应标注“更快、更稳、输出参数建议”
- 生成式路径应标注“更慢、可生成新图、需额外校验”
- 用户做出选择前，不得静默自动进入分析式或生成式流程

## 7.2 路径一：分析式风格迁移

流程：

```text
参考图
 + 当前图
   ↓
本地图像特征分析
   ↓
风格接近度评估
   ↓
调色参数建议
   ↓
用户确认后继续 RAW 工作流
```

目标：

- 快
- 稳
- 可解释
- 可编辑

该路径适合专业摄影场景，但是否使用，必须由用户明确选择。

## 7.3 路径二：生成式风格预览

生成式能力不再作为默认主路径，而是“用户主动选择的预览模式”。

流程：

```text
参考图
 + 当前图
   ↓
缩放到预览分辨率
   ↓
IP-Adapter 风格注入
   ↓
ControlNet 结构约束
   ↓
SDXL img2img
   ↓
结果校验
   ↓
生成预览图
```

该路径的目标不是直接替代 RAW 流程，而是回答：

```text
这个风格方向值不值得继续尝试
```

### 7.3.1 生成式预览的 UI 位置

生成式风格预览必须在主编辑工作区中完成，不允许隐藏在难以发现的次级区域。

推荐规则：

- 预览显示在主画布预览区
- 默认以“临时预览态”覆盖显示
- 支持切换为“原图 / 生成式预览”对比视图
- 在右侧面板显示当前任务状态、模式、耗时和是否通过结果校验

明确限制：

- 生成式预览不应直接替换当前原图工作区状态
- 用户未确认前，不得把生成式预览视为正式结果
- 用户必须显式执行“应用为衍生图”或“进入导出”后，才进入下一阶段

### 7.3.2 生成式预览的交互要求

生成式预览至少需要提供以下交互：

- 查看原图
- 查看生成式预览
- 原图与预览对比
- 放弃本次预览
- 应用为衍生图
- 进入高质量导出

### 7.3.3 进入生成式预览阶段的前置流程

从用户触发生成式预览，到真正进入 SDXL 推理之前，系统需要经过一段前置流程。

建议链路如下：

```text
用户点击生成式预览
  ↓
校验当前模式、参考图、当前图、基础参数
  ↓
检查服务是否 ready
  ↓
加载或复用预览分辨率图像
  ↓
生成或复用 control image
  ↓
生成或复用参考图风格编码结果
  ↓
组装预览推理参数
  ↓
进入生成式预览推理
```

这条链路本身是合理的，因为它对应了生成式预览真正需要的输入准备工作；但如果这些步骤没有前置裁剪、并行化和缓存复用，就会在“还没正式推理之前”先堆出明显的等待时间。

#### 前置流程中合理且必须存在的步骤

- 校验参考图、当前图和模式是否合法
- 确认服务 ready，而不是盲目发起任务
- 将原图缩放到预览分辨率，而不是直接拿大图进入预览链路
- 生成结构控制所需的 control image
- 生成风格注入所需的参考图编码结果
- 在推理前完成参数组装和版本绑定

#### 前置流程中容易出现的重复冗余

以下步骤如果在每次预览都重复执行，就会形成不必要的性能损耗：

- 重复读取原图和参考图
- 重复缩放到同一预览分辨率
- 重复生成同一张 control image
- 重复编码同一参考图风格信息
- 重复进行服务预热检查
- 在参数未变化时重复组装完全相同的推理任务

#### 前置流程中的主要性能瓶颈

进入预览阶段前，最容易形成瓶颈的不是 SDXL 推理本身，而是以下几段准备链路：

- 大图读取与缩放
- control image 生成
- 参考图风格编码
- 首次服务预热
- 参数准备与缓存命中失败

因此，预览阶段的首段优化重点必须放在“少做一次准备，胜过晚做一次推理”。

#### 文档要求的收敛策略

为了避免前置流程成为新的慢路径，必须满足：

- 预览图缩放结果可复用
- control image 可复用
- 参考图风格编码结果可复用
- 服务 ready 状态应由模型常驻层统一维护
- 参数组装结果应绑定输入签名，避免重复生成

如果以上任一环节未收敛，生成式预览即使推理步数较低，也仍然可能在进入推理前耗掉大量时间。

## 7.4 最终导出路径

只有在以下条件全部满足时，才允许走高质量导出链路：

- 预览效果已被用户确认
- GPU 状态正常
- 模型与依赖全部就绪
- 输出校验通过

如果无法满足以上条件，应阻止导出，而不是强行生成结果。

# 八、性能预算与交互策略

## 8.1 预览分辨率策略

生成式预览阶段统一限制输入分辨率：

- 长边 1024 为默认
- 长边 1280 为上限
- 默认不对全分辨率原图直接执行生成式推理

## 8.2 步数策略

预览模式：

- Steps：12 到 20
- 默认关闭 Refiner

导出模式：

- Steps：20 到 30
- Refiner 仅在用户主动开启时使用

## 8.3 大图策略

大图处理不再采用“默认逐 tile 完整扩散”的产品路径。

改为分层策略：

- 预览阶段：统一缩小后整图推理
- 导出阶段：仅对确认后的任务进入高分辨率处理
- 大图分块仅作为导出能力，不作为默认交互能力

## 8.4 超时策略

- 分析式风格迁移超过 10 秒，应视为异常并记录诊断信息
- 单次预览超过 30 秒，UI 给出明显告警
- 单次预览超过 2 分钟，默认判定为超时失败或需要用户明确确认继续等待
- 单次导出超过 2 分钟，需要用户明确确认继续等待
- 服务预热超过 20 秒，应提示用户服务未就绪，而不是长时间无反馈

# 九、受控的空间换时间策略

## 9.1 总体原则

允许使用“空间换时间”策略，但必须满足以下前提：

- 不破坏当前文档定义的产品边界
- 不绕过结果校验
- 不掩盖真实失败
- 不以缓存伪造成功结果

换句话说：

```text
缓存和常驻只能减少重复计算
不能替代真实推理
不能隐藏错误
不能制造过期结果
```

## 9.2 适用范围

### 分析式风格迁移

最适合采用空间换时间策略。

可缓存对象：

- 参考图风格特征
- 当前图基础统计信息
- 风格标签与场景标签
- 参数建议中间结果

目标：

- 将重复分析稳定压到 1 到 5 秒
- 对同一参考图的重复使用尽量做到秒开

### 生成式风格预览

允许采用受控缓存，但不能缓存最终高质量结果替代真实执行。

可缓存对象：

- 缩放后的预览图
- Control Image
- 参考图风格编码结果
- 低分辨率预览结果
- 任务级中间元数据

目标：

- 避免重复缩放、重复编码、重复控制图生成
- 优先保证预览路径稳定落入 SLA

### 高质量导出

只允许复用预览阶段的安全中间结果，不允许以过时缓存跳过关键导出计算。

可复用对象：

- 参考图分析结果
- 缩放策略和裁剪参数
- 预览阶段已验证通过的配置
- 已验证可复用的控制图与推理参数

## 9.3 推荐的缓存层

### A. 模型常驻层

必须支持以下组件常驻内存或显存：

- SDXL Base
- IP-Adapter
- ControlNet
- 可选 Refiner

要求：

- 服务启动完成后不重复加载主模型
- 模型预热结果必须有明确 ready 状态
- 模型失效时必须显式回报，不允许静默降级

### B. 内存缓存层

适用于短时高频复用对象：

- 参考图特征
- 当前图缩略统计
- control image
- 预览尺寸图像
- 最近一次预览结果

要求：

- 有数量上限
- 有大小上限
- 采用 LRU 或类似淘汰策略

### C. 磁盘缓存层

适用于跨会话复用对象：

- 参考图分析结果
- 预览阶段的中间元数据
- 已验证通过的低分辨率性能缓存结果

要求：

- 必须与输入文件签名绑定
- 必须支持失效
- 必须能清理
- 不得与 UI 临时预览文件混用

### D. 导出复用层

用于把预览阶段的安全结果复用于导出阶段。

可复用内容：

- 用户确认过的 preset
- 预览期生成的 control image
- 预览期确认有效的推理参数

限制：

- 不得直接把低分辨率预览图当作高质量导出结果
- 不得跳过导出阶段必要的质量校验

## 9.4 缓存 Key 设计

所有缓存必须绑定输入签名，至少包括：

- 参考图路径
- 当前图路径
- 文件大小
- 文件修改时间
- 模式
- preset
- 关键推理参数
- 版本号

建议规则：

```text
只要输入文件变化、参数变化、模型版本变化、算法版本变化
缓存就必须失效
```

## 9.5 必须记录的状态

为了保证可追踪性，每个缓存对象至少记录：

- 创建时间
- 输入签名
- 生成版本
- 来源路径
- 是否通过结果校验

未通过结果校验的对象不得提升为可复用缓存。

## 9.6 允许的优化手段

- 模型常驻与预热
- 参考图特征缓存
- 控制图缓存
- 低分辨率预览结果缓存
- 参数与中间元数据缓存
- 导出阶段复用预览阶段的安全配置

## 9.7 明确禁止的优化手段

- 用旧缓存结果替代当前真实推理
- 在输入变化后继续复用旧缓存
- 用缓存掩盖 IP-Adapter 或 ControlNet 失效
- 将失败任务结果写入缓存
- 将未校验结果写入缓存
- 无上限缓存所有 tile、latent 或中间图像

## 9.8 与 SLA 的关系

受控的空间换时间策略服务于 SLA，但不能破坏 SLA 语义。

具体要求：

- 分析式风格迁移优先通过缓存和特征复用满足 1 到 5 秒目标
- 生成式风格预览优先通过模型常驻和预览缓存满足 10 到 30 秒目标
- 高质量导出只能复用安全中间结果，不能把导出变成“伪快”

## 9.9 工程实施建议

### Phase 2 应落地

- 模型常驻
- 参考图特征缓存
- control image 缓存
- 预览图缩放缓存
- 最近结果缓存

### Phase 3 应落地

- 导出阶段配置复用
- 导出阶段安全中间结果复用
- 磁盘缓存与失效机制

### Phase 4 才考虑

- 更细粒度的中间结果复用
- 分块级安全缓存
- 更复杂的跨任务复用系统

# 十、质量底线与结果校验

## 10.1 结果校验是强制步骤

生成式输出必须经过自动校验。

至少检查：

- 平均亮度是否异常低
- RGB 三通道是否过度趋同
- 是否存在大面积纯黑区域
- 是否存在 tile 级失败

## 10.2 禁止的错误处理方式

以下处理方式明确禁止：

- tile 失败后使用黑块继续拼接
- IP-Adapter 失效后继续当作正常风格迁移运行
- 结果异常时仍保存为成功输出
- 无差别多次强制值域归一化

## 10.3 正确的错误处理方式

- 任意 tile 失败则整单失败
- 核心风格注入失败则明确报错
- 输出校验失败则拒绝保存最终结果
- 将失败原因显示给 UI，而不是吞掉异常

## 10.4 质量守门模块要求

结果校验能力必须从风格迁移主流程中拆出，作为独立的 `quality guard` 模块存在。

这样做的目的不是增加复杂度，而是确保：

- 预览与导出共享同一套质量判断标准
- 校验逻辑可以独立迭代，不污染推理主流程
- 当生成式模块临时下线时，质量守门模块也可以整体拔出
- 不需要改动现有预览与导出主基建即可完成接入

# 十一、色彩与画质策略

## 11.1 输出格式定义

输出格式仍保持：

```text
16-bit RGB 衍生图像
```

但必须同时写清楚：

- 这不是 RAW
- 这不是对原 RAW 的无损替代
- 这是生成式结果文件

### 11.1.1 生成式导出格式选择策略

生成式导出结果应允许用户在正式导出阶段选择输出格式，但格式支持必须符合当前架构边界。

当前建议：

- 支持 `JPG`
- 支持 `PNG`
- 支持 `TIFF`
- 是否支持 `WEBP / JXL` 取决于现有导出基建能力，可作为扩展项

对于 `DNG`，必须明确：

- 当前架构下不应把生成式结果默认承诺为 `DNG` 输出
- 因为生成式结果本质上是 `16-bit RGB 衍生图`，不是 RAW 传感器数据
- 即使未来提供 `.dng` 容器形式输出，也不得在产品语义上把它描述成“RAW 恢复”或“真实 RAW 导出”

因此，当前文档的正式建议是：

- `JPG / PNG / TIFF` 可作为生成式结果导出格式
- `DNG` 不作为当前默认承诺能力
- 如果后续研究 `DNG` 包装输出，必须单独立项、单独验证、单独说明语义边界

### 11.1.2 生成式结果的存储与归属

生成式结果必须区分“预览结果”和“正式结果”。

#### 预览结果

- 只允许写入临时预览缓存区
- 不进入正式素材库
- 不覆盖原图
- 允许在会话结束或缓存清理时删除

这里的“预览结果”专指 UI 展示使用的临时预览文件，不等同于性能优化所使用的跨会话磁盘缓存对象。

#### 用户确认后的衍生结果

- 写入项目衍生资源目录
- 在项目中归属于原图的 AI 衍生版本
- 不替代原图，不修改原 RAW
- 必须与原图建立可追踪关联

#### 高质量导出结果

- 写入正式导出目录
- 与普通导出结果并列管理
- 必须保留来源信息：原图、参考图、模式、时间、版本

### 11.1.3 项目内显示规则

在项目界面中，16-bit RGB 衍生图必须以“衍生结果”身份出现，而不是伪装成原图。

必须满足：

- 原图与衍生图可区分
- 原图与衍生图可互相跳转
- 原图下可以看到其对应的 AI 衍生版本
- 用户可以删除衍生结果而不影响原图

### 11.1.4 资产归属模块要求

生成式结果的存储与归属管理必须作为独立的 `asset registry` 模块实现。

这个模块负责：

- 为预览结果、衍生结果、正式导出结果分配身份
- 建立原图与生成式结果之间的映射关系
- 决定哪些对象进入缓存区，哪些进入项目资源目录，哪些进入正式导出目录

明确禁止：

- 直接把生成式结果硬写入现有原图 sidecar 语义
- 直接把生成式衍生图伪装成普通原图条目
- 在没有归属登记的情况下将结果暴露给正式项目列表

## 11.2 色彩对齐策略

色彩对齐不再作为“文档已完成能力”直接宣称。

只有在以下条件满足后，才可以重新启用：

- 实际执行路径已接入
- 输出不再引入黑图/灰图风险
- 已通过针对肤色、高光、阴影的回归测试

在重新验证前，色彩对齐仅作为候选增强模块记录，不作为当前上线能力。

## 11.3 RAW 融合策略

RAW 融合仍然是长期有价值方向，但当前不应继续作为已完成能力对外描述。

重新启用前必须满足：

- 清楚定义融合输入与输出的色彩空间
- 验证高光与阴影保护的有效性
- 明确融合后仍然是 RGB 衍生图，而非 RAW 恢复

# 十三、产品与工程一致性要求

## 13.1 UI 必须明确表达

- 当前模式是分析式还是生成式
- 生成式输出的是新图，不是 RAW 本体
- 当前运行的是预览还是导出
- 当前是否发生了 fallback
- 当前显示的是原图、临时预览还是已确认的衍生图

## 13.2 健康检查必须真实

健康检查返回的 capabilities 只能包含真正可用的能力。

如果色彩对齐、RAW 融合、Refiner 未真实接入，则不得标记为 ready 能力。

## 13.3 文档必须真实

后续所有文档都必须遵循：

```text
只写已经真实接入并验证通过的能力
不写理想目标冒充当前状态
```

## 13.4 模块边界必须真实

所有新增 AI 功能都必须以“可拔插模块”身份存在，而不是隐藏在现有基建内部。

至少要满足：

- 模块目录、接口、依赖边界清晰可见
- 模块启用和禁用不影响现有非 AI 编辑功能
- 模块与现有预览、导出、资源管理之间只通过显式接口连接
- 模块内部状态不得泄漏成现有基建的隐式全局状态
- AI 修图、AI 选图、风格迁移三类能力必须遵循同一套边界规则
- 上游基建同步时，AI 模块应尽量保持低冲突合并成本

## 13.5 多语言与代码质量规范

所有我们自己集成的功能，必须同时满足以下要求：

- 必须接入项目现有多语言体系
- 不允许在组件、面板、弹窗、任务提示、错误提示中直接写死单语言文案
- 新增前端代码必须严格使用 TypeScript
- 不允许新增 `.js`、`.jsx` 版本实现去绕过现有 TS 规范
- TypeScript 类型定义必须完整，避免使用无约束的 `any` 作为常态实现方式
- 新增模块必须通过统一的代码质量检查，不得以“实验功能”为理由跳过规范

这条规则适用于：

- AI 修图
- AI 选图
- 风格迁移
- 后续所有新增 AI 能力

# 十四、阶段性实施路线

## Phase 1：止血版本

目标：

- 终止黑图、灰图、伪成功输出
- 将默认预览时间压回到 30 秒级，并确保最迟不超过 2 分钟

工作项：

- 移除 tile 失败补黑块逻辑
- 核心依赖失败时立即报错
- 关闭默认大图分块生成式路径
- 统一为预览分辨率整图推理
- 增加结果亮度与色彩校验

## Phase 2：可用版本

目标：

- 形成稳定的“分析式主功能 + 生成式预览”产品结构

工作项：

- 明确 UI 模式边界
- 增加任务取消、超时、错误回传
- 优化默认 preset
- 规范生成图命名与结果目录
- 建立模型常驻和基础缓存层
- 拆出 `preview adapter`、`quality guard`、`task orchestrator` 三个独立模块
- 通过接口方式接入现有主画布、预览 worker 和设置系统

## Phase 3：导出版本

目标：

- 在预览稳定基础上，增加可控的高质量导出能力

工作项：

- 导出阶段高分辨率处理
- 受控启用大图分块
- 可选 Refiner
- 导出前质量校验
- 增加导出阶段复用与磁盘缓存失效机制
- 拆出 `export adapter` 与 `asset registry` 模块
- 通过模块引用对接现有导出链路和项目资源管理

## Phase 4：增强版本

目标：

- 在稳定性达标后，逐步恢复高级画质模块

工作项：

- 验证后重新接入色彩对齐
- 验证后重新接入 RAW 融合
- 建立回归测试集
- 在保持模块边界不变的前提下迭代增强质量能力

# 十五、最终结论

新的结论不再是“工业级最终落地方案”，而是：

```text
这是一次从失败实现回到真实用户需求的架构重构
```

本版本的价值在于：

- 明确承认当前实现不可直接上线
- 明确把“30 分钟等待”和“黑图灰图”视为硬性失败
- 明确将风格迁移拆成分析式主路径与生成式预览路径
- 明确以可用性、稳定性、一致性取代空泛的高质量叙事
- 明确 AI 修图、AI 选图、风格迁移都必须以无侵入、可拔插、模块引用方式接入现有基建
- 明确这样做的目标是持续迭代 AI 功能，同时持续同步上游开源基建，避免长期冲突

# 十六、一句话总结

```text
RapidRAW 风格迁移不是先追求“最强模型”
而是先建立“30 秒级预览、预览最迟 2 分钟内返回、导出超时需明确确认、结果不出事故、工作流语义清晰、并且所有 AI 功能都可无侵入接入现有基建”的专业能力
```
