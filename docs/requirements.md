# 需求文档

## 简介

本文档规定了 RapidRAW 图像编辑器中**相机对焦区域标注功能**的需求。该功能使摄影师能够通过从 RAW 文件中提取对焦点元数据并将其作为叠加层渲染在图像画布上,来可视化相机捕获的实际对焦区域。这是一个元数据驱动的功能,不使用 AI 推理,仅支持 RAW 文件格式。

该功能满足了摄影师了解相机在拍摄时对焦位置的需求,这对于检查对焦准确性、理解自动对焦行为和改进拍摄技术非常有价值。

## 术语表

- **对焦区域提取器 (Focus_Region_Extractor)**: 负责解析 RAW 文件元数据并提取对焦点信息的后端 Rust 组件
- **对焦叠加层渲染器 (Focus_Overlay_Renderer)**: 负责将对焦区域作为视觉叠加层渲染在图像画布上的前端 React/Konva 组件
- **对焦适配器 (Focus_Adapter)**: 基于 trait 的抽象,定义了从不同相机制造商提取对焦元数据的相机特定逻辑
- **归一化坐标 (Normalized_Coordinates)**: 对焦区域坐标表示为 0.0 到 1.0 之间的浮点值,相对于完整图像尺寸
- **变换管道 (Transform_Pipeline)**: 应用于图像和对焦区域的几何变换序列(EXIF 旋转、用户旋转、翻转、裁剪)
- **制造商备注 (MakerNotes)**: 特定于每个相机制造商的专有 EXIF 元数据字段,包含对焦点信息
- **对焦类型 (Focus_Kind)**: 对焦区域的类型(点、区域、人脸、眼睛)
- **主对焦点 (Primary_Focus_Point)**: 相机或摄影师选择的主要对焦点
- **RAW 文件 (RAW_File)**: 未处理的图像文件格式(例如 ARW、CR2、NEF),包含传感器数据和完整元数据

## 需求

### 需求 1: 从 RAW 文件提取对焦元数据

**用户故事:** 作为摄影师,我希望系统从我的 RAW 文件中提取对焦点元数据,以便我可以看到相机在拍摄时的对焦位置。

#### 验收标准

1. 当加载 RAW 文件时,对焦区域提取器应使用现有的 `kamadak-exif` 和 `rawler` 库解析文件
2. 当成功提取对焦元数据时,对焦区域提取器应返回具有归一化坐标(0.0-1.0 范围)的对焦区域列表
3. 当不支持相机制造商时,对焦区域提取器应返回错误消息,指示不支持的制造商和型号
4. 当 RAW 文件不包含对焦元数据时,对焦区域提取器应返回空列表
5. 对焦区域提取器应在 50 毫秒内完成对 100MB 以下文件的提取
6. 对于所有提取的对焦区域,坐标应在任何变换之前相对于完整传感器尺寸进行归一化

### 需求 2: 支持多个相机制造商

**用户故事:** 作为使用不同相机品牌的摄影师,我希望系统支持来自 Sony、Canon 和 Nikon 相机的对焦元数据,以便无论我使用哪个相机品牌都可以使用此功能。

#### 验收标准

1. 对焦区域提取器应实现一个 Focus_Adapter trait,定义相机特定提取逻辑的通用接口
2. 对焦区域提取器应提供 Sony_Adapter 实现,从 Sony RAW 文件中提取对焦元数据
3. 当启用 Canon 支持时,对焦区域提取器应提供 Canon_Adapter 实现
4. 当启用 Nikon 支持时,对焦区域提取器应提供 Nikon_Adapter 实现
5. 当选择适配器时,对焦区域提取器应从 EXIF `Make` 字段匹配相机制造商
6. 当没有适配器匹配相机制造商时,对焦区域提取器应返回包含不支持制造商名称的错误

### 需求 3: 表示对焦区域数据

**用户故事:** 作为开发人员,我希望有一个标准化的对焦区域数据结构,以便前端可以一致地渲染它们,无论相机制造商如何。

#### 验收标准

1. 对焦区域提取器应定义包含归一化 x、y、width、height 坐标的 FocusRegion 结构
2. FocusRegion 结构应包含 `kind` 字段,指示对焦类型(Point、Area、Face、Eye)
3. FocusRegion 结构应包含 `is_primary` 布尔字段,指示它是否为主对焦点
4. 对焦区域提取器应将 FocusRegion 数据序列化为 JSON 以传输到前端
5. 对于所有对焦区域,归一化坐标应相对于任何用户变换之前的原始传感器尺寸

### 需求 4: 通过 Tauri 命令提供对焦区域数据

**用户故事:** 作为前端开发人员,我希望通过 Tauri 命令请求对焦区域数据,以便在 UI 中显示对焦叠加层。

#### 验收标准

1. 对焦区域提取器应公开接受文件路径参数的 `get_focus_regions` Tauri 命令
2. 当调用 `get_focus_regions` 命令时,对焦区域提取器应返回 FocusRegion 对象的 JSON 数组
3. 当提取失败时,`get_focus_regions` 命令应返回描述失败原因的错误字符串
4. `get_focus_regions` 命令应通过 `parse_virtual_path` 提取源文件路径（返回 `(source_path, sidecar_path)` 元组，忽略 sidecar）来处理虚拟副本路径
5. 当多次请求同一文件时,对焦区域提取器应缓存解析结果以避免冗余解析

### 需求 5: 切换对焦区域显示

**用户故事:** 作为摄影师,我希望能够打开和关闭对焦区域的显示,以便我可以比较有无对焦叠加层的图像。

#### 验收标准

1. 编辑器工具栏应提供带有目标图标的"显示对焦区域"按钮
2. 当单击按钮时,编辑器工具栏应切换 `showFocusAreas` 状态
3. 当 `showFocusAreas` 状态为 true 时,按钮应显示为活动/高亮外观
4. 当 `showFocusAreas` 状态为 false 时,按钮应显示为非活动外观
5. 按钮应显示工具提示,在非活动时指示"显示对焦区域 (Shift+F)",在活动时指示"隐藏对焦区域 (Shift+F)"
6. 当用户按下 Shift+F 时,系统应切换 `showFocusAreas` 状态

### 需求 6: 启用显示时加载对焦区域

**用户故事:** 作为摄影师,我希望在启用显示时自动加载对焦区域,这样我就不必手动触发加载过程。

#### 验收标准

1. 当 `showFocusAreas` 更改为 true 时,系统应使用当前图像路径调用 `get_focus_regions` Tauri 命令
2. 当成功加载对焦区域时,系统应将它们存储在 `focusRegions` 状态中
3. 当对焦区域加载失败时,系统应显示带有错误消息的 toast 通知
4. 当 `showFocusAreas` 更改为 false 时,系统应清除 `focusRegions` 状态
5. 当所选图像更改时,如果 `showFocusAreas` 为 true,系统应重新加载对焦区域
6. 对于不支持的相机,toast 通知应包含有关提交样本文件以添加支持的信息

### 需求 7: 变换对焦坐标

**用户故事:** 作为摄影师,我希望对焦区域跟随我的图像变换(旋转、翻转、裁剪),以便在编辑图像时叠加层保持准确。

#### 验收标准

1. 对焦叠加层渲染器应实现 `transformFocusPoint` 函数,将所有活动变换应用于对焦坐标
2. `transformFocusPoint` 函数应首先应用 EXIF 方向旋转(基于 `orientationSteps` 的 0°、90°、180°、270°)
3. `transformFocusPoint` 函数应其次应用用户旋转(来自 `adjustments.rotation` 的任意角度)
4. `transformFocusPoint` 函数应第三应用水平翻转(如果 `adjustments.flipHorizontal` 为 true)
5. `transformFocusPoint` 函数应第四应用垂直翻转(如果 `adjustments.flipVertical` 为 true)
6. `transformFocusPoint` 函数应第五应用裁剪偏移(从对焦坐标中减去裁剪 x/y)
7. `transformFocusPoint` 函数应最后应用画布缩放(乘以当前缩放比例)
8. 对于所有变换,函数应将坐标精度保持在预期位置的 2 像素内

### 需求 8: 在画布上渲染对焦叠加层

**用户故事:** 作为摄影师,我希望看到对焦区域渲染为图像上的彩色矩形,以便我可以直观地识别相机对焦的位置。

#### 验收标准

1. 当 `showFocusAreas` 为 true 且 `focusRegions` 不为空时,对焦叠加层渲染器应为每个对焦区域渲染一个 Konva Rect
2. 当对焦区域是主对焦点时,对焦叠加层渲染器应使用红色描边 (#ef4444) 渲染它
3. 当对焦区域类型为 Face 时,对焦叠加层渲染器应使用绿色描边 (#22c55e) 渲染它
4. 当对焦区域类型为 Eye 时,对焦叠加层渲染器应使用蓝色描边 (#3b82f6) 渲染它
5. 当对焦区域类型为 Point 时,对焦叠加层渲染器应使用虚线描边模式渲染它
6. 当对焦区域类型为 Area 时,对焦叠加层渲染器应使用实线描边渲染它
7. 对焦叠加层渲染器应使用 2 像素描边宽度渲染所有对焦叠加层
8. 对焦叠加层渲染器应使用黑色阴影(模糊: 4px,不透明度: 0.6)渲染所有对焦叠加层,以在浅色背景上可见
9. 对焦叠加层渲染器应在所有对焦叠加层形状上设置 `listening={false}` 以防止交互

### 需求 9: 优雅地处理不支持的相机

**用户故事:** 作为使用不支持相机的摄影师,我希望收到关于为什么无法显示对焦区域的清晰反馈,以便我了解限制并知道如何帮助添加支持。

#### 验收标准

1. 当由于不支持的相机导致对焦提取失败时,系统应显示带有错误消息的 toast 通知
2. toast 通知应至少保持可见 5 秒
3. toast 通知应包含相机制造商和型号名称
4. toast 通知应包含鼓励用户提交样本文件以添加支持的消息
5. 当对焦提取失败时,`showFocusAreas` 按钮应保持功能正常,不进入错误状态

### 需求 10: 通过缓存优化性能

**用户故事:** 作为浏览多张图像的摄影师,我希望在返回到先前查看的图像时快速加载对焦区域数据,以便界面保持响应。

#### 验收标准

1. 对焦区域提取器应为解析的对焦区域数据实现缓存
2. 缓存键应从文件路径派生
3. 当请求缓存文件的对焦区域时,对焦区域提取器应在 10 毫秒内返回缓存数据
4. 当文件被修改时(基于文件修改时间戳),对焦区域提取器应使缓存条目无效
5. 缓存应具有最大大小限制以防止无限制的内存增长

### 需求 11: 支持不同的对焦类型

**用户故事:** 作为摄影师,我希望区分不同类型的对焦区域(单点、区域、人脸检测、眼睛检测),以便我可以了解使用了什么自动对焦模式。

#### 验收标准

1. 对焦区域提取器应将对焦区域分类为四种类型之一: Point、Area、Face 或 Eye
2. 当相机报告单点对焦时,对焦区域提取器应将类型设置为 Point
3. 当相机报告区域或区域对焦时,对焦区域提取器应将类型设置为 Area
4. 当相机报告人脸检测对焦时,对焦区域提取器应将类型设置为 Face
5. 当相机报告眼睛检测对焦时,对焦区域提取器应将类型设置为 Eye
6. 当无法确定对焦类型时,对焦区域提取器应默认为 Area

### 需求 12: 保持与现有叠加层的视觉一致性

**用户故事:** 作为用户,我希望对焦区域叠加层与现有编辑器叠加层(构图指南、蒙版)无缝集成,以便界面感觉连贯。

#### 验收标准

1. 对焦叠加层渲染器应在与其他图像叠加层相同的 Konva Layer 中渲染对焦叠加层
2. 对焦叠加层渲染器应在图像上方但在交互元素(蒙版、裁剪手柄)下方渲染对焦叠加层
3. 对焦叠加层渲染代码应遵循 ImageCanvas.tsx 中现有叠加层代码的相同模式
4. 对焦区域切换按钮应遵循 EditorToolbar.tsx 中其他工具栏按钮的相同样式模式

### 需求 13: 处理坐标变换中的边界情况

**用户故事:** 作为开发人员,我希望坐标变换逻辑正确处理边界情况,以便对焦叠加层在所有场景中保持准确。

#### 验收标准

1. 当图像没有应用裁剪时,`transformFocusPoint` 函数应使用完整图像尺寸
2. 当图像具有基于百分比的裁剪时,`transformFocusPoint` 函数应在应用偏移之前将裁剪百分比转换为像素值
3. 当图像具有基于像素的裁剪时,`transformFocusPoint` 函数应直接使用裁剪像素值
4. 当图像旋转 90° 或 270° 时,`transformFocusPoint` 函数应交换宽度和高度尺寸
5. 当对焦区域落在裁剪区域之外时,对焦叠加层渲染器仍应渲染它(它将被画布边界裁剪)
6. 当画布缩放为 0 或负数时,`transformFocusPoint` 函数应将其视为 1.0 以防止渲染错误

### 需求 14: 提供开发人员友好的错误消息

**用户故事:** 作为调试对焦提取问题的开发人员,我希望有详细的错误消息和日志记录,以便我可以快速诊断问题。

#### 验收标准

1. 当对焦提取失败时,对焦区域提取器应使用 `log::warn!` 或 `log::error!` 宏记录错误
2. 错误日志应包含相机制造商和型号
3. 错误日志应包含具体的失败原因(例如,"未找到 MakerNotes","不支持的格式")
4. 当无法读取 RAW 文件时,对焦区域提取器应返回指示文件读取失败的错误消息
5. 当元数据解析失败时,对焦区域提取器应返回指示解析失败的错误消息

### 需求 15: 确保前后端通信中的类型安全

**用户故事:** 作为开发人员,我希望有对焦区域数据的 TypeScript 类型定义,以便我可以在编译时捕获类型错误。

#### 验收标准

1. 前端应定义与 Rust 结构匹配的 TypeScript 接口 `FocusRegion`
2. `FocusRegion` 接口应包含字段: x、y、width、height(全部为 number)、kind(字符串联合)、is_primary(boolean)
3. 前端应将 TypeScript 类型 `FocusKind` 定义为 "point" | "area" | "face" | "eye" 的联合
4. `invoke<FocusRegion[]>` 调用应使用泛型类型参数以确保类型安全
5. 如果使用不正确的属性名称访问对焦区域数据,TypeScript 编译器应报告错误
