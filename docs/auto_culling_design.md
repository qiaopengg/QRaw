# QRaw AI 智能选图（Auto Culling）技术方案设计文档

## 1. 项目背景与目标

### 1.1 问题定义
摄影师在单次拍摄任务（如婚礼、活动、人像写真）中往往会产生数以千计的 RAW 照片。其中包含大量连拍冗余、失焦废片、曝光失误或闭眼/表情不佳的照片。
传统的手工筛选（Culling）极其耗时。本项目旨在利用 QRaw 现有的高性能本地计算基建（Rust + ONNX Runtime + WebGPU），实现一个**全自动的 AI 智能选图系统**。

### 1.2 核心目标
系统不再尝试用一个“玄学黑盒模型”去理解美，而是通过**“多维度基础特征提取 + 规则系统融合 + 连拍去重降维”**的工程化手段，为每张照片输出一个量化的**综合评分（1-5 星）**，帮助用户一键过滤废片，选出各场景的最优照片。

---

## 2. 架构设计：多模型组合与规则融合

AI 选图本质上是一个多阶段的流水线（Pipeline）。系统将分为三个主要阶段：特征提取、全局相似度分析、综合打分与评级。

### 2.1 整体数据流向
```text
[RAW 图像目录] 
      ↓ (并行解码与缩略图生成)
[特征提取层 (并行)]
  ├── 清晰度检测 (Laplacian 方差)
  ├── 曝光与色彩健康度 (基于现有的 perform_auto_analysis)
  ├── 语义与美学特征 (基于 CLIP Embedding)
  └── 人脸与表情检测 (待引入轻量级 ONNX 模型)
      ↓ (特征聚合)
[全局分析层 (内存聚合)]
  └── 连拍去重与相似度聚类 (基于 CLIP Embedding 余弦相似度 + 拍摄时间)
      ↓ (规则引擎)
[打分与评级层]
  ├── 废片直接淘汰 (1星)
  ├── 连拍组内优选 (同组最优 4/5 星，其余降级)
  └── 独立好片提权 (3-5星)
      ↓
[持久化层]
  └── 写入 .rrdata (ImageMetadata.rating) -> 前端 UI 更新照片墙星级
```

## 3. 核心技术模块设计与落地细节

### 3.1 清晰度检测（Blur Detection）
**目标**：判断照片是否失焦或存在严重动态模糊。
**方案**：无需重型 AI 模型，使用经典 CV 算法。
*   **算法**：拉普拉斯方差（Variance of Laplacian）。
*   **原理**：计算图像灰度图的拉普拉斯算子，其方差代表了图像中高频边缘信息的丰富程度。方差越低，图像越模糊。
*   **QRaw 落地**：
    *   在 [image_processing.rs](file:///Users/qiaopeng/Desktop/owner/QRaw/src-tauri/src/image_processing.rs) 中新增纯 Rust 实现，利用 `rayon` 并行处理缩略图的 `Luma` 通道。
    *   **复杂度**：极低。

### 3.2 图像质量评分（Exposure & Color Health）
**目标**：判断照片是否存在严重的曝光失误（死黑/死白）或色彩异常（严重偏色）。
**方案**：白嫖 QRaw 现有基建。
*   **QRaw 落地**：
    *   直接复用 [image_processing.rs:L2534](file:///Users/qiaopeng/Desktop/owner/QRaw/src-tauri/src/image_processing.rs#L2534) 中的 `perform_auto_analysis` 函数。
    *   该函数已经实现了对图像高光、阴影、直方图分布、饱和度的精确计算。
    *   **规则**：如果该函数返回的 `exposure` 补偿值极大（如 > +3.0 或 < -3.0），或者 `shadows`/`highlights` 需要极端的恢复，则判定原图曝光质量差，进行扣分。

### 3.3 去重与连拍筛选（Duplicate / Burst Grouping）
**目标**：识别出连续拍摄的相似照片，将其归为一组，并在组内进行“内卷”选优，大幅减少相似照片数量。
**方案**：RAW+JPEG 关联前置 + 基础画质预过滤 + 时间窗约束 + 混合特征图聚类。
*   **QRaw 落地**：
    *   复用 [ai_processing.rs](file:///Users/qiaopeng/Desktop/owner/QRaw/src-tauri/src/ai_processing.rs) 中已集成的 `CLIP` 模型。
    *   **聚类逻辑**：
        1. **同名异构过滤（RAW+JPEG 严格堆叠）**：在提取特征前，首先扫描目录进行资产绑定。**严格绑定边界**：仅当文件名完全一致（不含后缀，如 `IMG_001`）且 EXIF 拍摄时间差 `< 1s` 时才绑定。防止将用户导出的 `IMG_001_edit.JPG` 错误绑定。系统默认以 RAW 文件为主体参与后续选图，副文件直接继承主文件的评分和标签，绝不进入连拍去重比较。
        2. **Stage 0: 超快过滤与性能熔断**：在调用昂贵的 CLIP 模型前，先进行基础画质判定。如果照片被判定为极端模糊（Laplacian 极低）或极端曝光失误，**不提取 CLIP Embedding，但也不强行抛弃**。而是直接赋以极低基础分，并在 pipeline 中绕过重型网络（Skip Heavy Model），让其自然沉底到 1 星或 2 星。这既节省了算力，又避免了“刻意抖动抓拍/情绪照片”被直接销毁。
        3. **Stage 1: 时间窗与容量约束 (性能熔断)**：读取剩余有效资产的 EXIF 拍摄时间进行分组。为防止 O(n²) 聚类耗尽内存带宽，强制限制单个时间窗口的最大照片数（如 `Window_Size <= 50`）。
        4. **Stage 2: 混合特征图聚类 (Graph Clustering)**：在受限时间窗内，计算两两图片的相似度。**核心工程约束**：所有 CLIP Embedding 在参与点乘前，**必须进行 L2 Normalization（归一化为单位向量）**，否则 Cosine 相似度阈值将完全失效。基于相似度构建连通图（Union-Find）。
        **核心防污染与动作约束**：为了防止动作被错误压平或长链污染，在加入 Cluster 时，该节点必须与当前 Cluster Center 的余弦相似度 `≥ 0.92`，**且任意两连通节点的拍摄间隔必须 `Δt ≤ 1.5s`（时间密度优先级 > 相似度）**，否则强制拆分为新组。
    *   **规则**：连拍组构建完成后，选出综合得分最高的一张作为该组的**封面图 (Best Image)**。前端 UI 应当基于 `group_id` 和 `is_cover` 字段来实现“折叠连拍”，而不是依赖星级。

### 3.4 构图与美学评分（Composition & Aesthetics）
**目标**：评估照片的构图是否合理，是否具有美感。
**方案**：引入轻量级专用美学打分模型（Aesthetic Quality Model）或基于简单启发式规则。
*   **QRaw 落地**：
    *   **避坑指南（模型风险）**：像 NIMA 或 AVA 这样基于公开数据集训练的美学模型，往往带有强烈的“互联网审美偏见（高对比度、高饱和度、糖水片）”。它们在判断特定题材（如婚礼纪实、商业黑白人像、特定影调抓拍）时，经常会与专业摄影师的审美发生冲突，导致大量用户投诉“系统不懂美”。
    *   **新方案（第一版降级与剥离）**：
        1. **MVP 阶段**：完全不引入神经网络美学模型，只依赖客观质量（清晰度、曝光）和主观人脸质量（表情）。如果必须评估构图，使用简单的启发式规则（Heuristics），如：人脸是否位于中心/三分线、人脸是否被画面边缘裁切。
        2. **后续阶段**：如果引入 NIMA 等美学模型，其权重必须严格控制在 `w_aes <= 0.2`，绝不能让它主导最终打分。

### 3.5 人脸与表情评分（Facial Expression & Blink）*【需新增】*
**目标**：检测人脸，判断是否闭眼、表情是否自然（核心刚需）。
**方案**：引入级联的轻量级 ONNX 模型（人脸检测 + 眼睛状态分类）。
*   **QRaw 落地**：
    *   **避坑指南 1（5点局限性）**：轻量级人脸检测模型（如 RetinaFace）通常只输出 5 个关键点，这 5 个点只够做人脸对齐，绝对不够稳定判断闭眼（EAR）。
    *   **避坑指南 2（架构一致性）**：拒绝 MediaPipe FaceMesh 的核心原因并非仅是“Rust 绑定困难”，而是因为它自带庞大的 Pipeline Graph Runtime。这会破坏 QRaw 当前统一且轻量的 `ort` (ONNX Runtime) 推理架构。必须寻找纯 ONNX 格式的级联模型。
    *   **避坑指南 3（多人脸干扰）**：合照中往往有许多背景人脸或被遮挡的人脸，如果不加限制地全部算分，系统评分会极其混乱。**强制规则**：必须基于 `face_area / image_area` 排序，只提取占比最大的 **Top N（如 N=3）主体人脸**，并且面积 `< 1%` 的背景人脸必须被忽略。
    *   **新方案（两阶段原生 ONNX）**：
        1. **Stage 1 (检测)**：使用 RetinaFace 或 YOLOv8-Face 框出人脸。
        2. **Stage 2 (分类)**：过滤出主体人脸，裁剪出眼部/脸部区域，送入一个极轻量级的分类网络（如 MobileNetV3 训练的 Blink/Open-Eye Classifier），直接输出闭眼概率和微笑概率。
    *   **评分逻辑 (多人脸木桶效应)**：
        1. 遍历 Top N 个主体人脸。**只要其中有 1 个或以上的主体人脸闭眼概率 > 阈值** -> 立即施加严重惩罚并打上 `{"reason": "有人闭眼"}` 标签（婚礼/合影刚需）。
        2. 若无闭眼，取 Top N 人脸质量的平均分。
        3. 微笑概率 -> 辅助加分。
        4. 若无主体人脸，则跳过表情打分。

## 4. 评分融合与排序规则（Scoring & Fusion）

在完成特征提取后，系统进入规则引擎进行融合。我们定义总分为 `Total_Score` (0-100)，最终映射为 1-5 星。

### 4.1 评分归一化与加权公式 (Normalization & Weighting)
**核心避坑**：不同特征的量纲完全不同（如拉普拉斯方差为 `0~∞`，曝光偏移为 `-5~+5`，概率为 `0~1`）。直接加权会导致某一维度（如方差）数值过大从而完全主导总分。
**必须的数学处理**：在加权前，所有子分数必须经过数学映射到 `[0.0, 1.0]` 的标准区间。

```rust
// 伪代码示例：各项指标非线性归一化到 0.0~1.0
// 1. 清晰度：使用非线性映射（如对数或指数变换），将绝对物理量映射为感知质量。
// 【工程可复现性约束】：k 必须是一个基于缩略图分辨率归一化的常数（如 k = const / (width * height)），防止不同设备/分辨率下结果漂移。
Norm_Blur = 1.0 - exp(-k * blur_variance)

// 2. 曝光健康度：使用指数衰减（非线性），因为人眼对曝光的容忍度是非线性的（偏差1档可接受，偏差3档即废片）
Norm_Exposure = exp(-0.8 * abs(Exposure_Offset))

// 3. 美学分：线性映射 (MVP阶段建议废弃此项)
Norm_Aesthetic = Aesthetic_Score / 10.0

// 4. 面部质量：加权概率
Norm_Face = (1.0 - Blink_Prob) * 0.7 + Smile_Prob * 0.3

// 动态权重策略 (基于摄影场景/Profile)
let has_large_face = max_face_area / image_area > 0.05; // 明确定义大脸阈值
let (w_blur, w_exp, w_aes, w_face) = if has_large_face {
    (0.35, 0.20, 0.0, 0.45) // 人像优先 (收敛 Face 权重，防止 AI 过度偏好笑脸而忽略画质)
} else {
    (0.60, 0.40, 0.0, 0.0)  // 风光/静物优先
};

Total_Score = 
    (Norm_Blur * w_blur) +
    (Norm_Exposure * w_exp) +
    (Norm_Aesthetic * w_aes) +
    (Norm_Face * w_face)
```

### 4.2 评星决策树 (Decision Tree)
1. **严重缺陷过滤（柔性扣分而非硬淘汰）**：
   * 不采用“闭眼直接 1 星”这种生硬规则（如艺术模糊、大合照中单人闭眼）。
   * 如果 `Norm_Blur` 极低，给予大幅扣分（如总分乘以 0.5）。
   * 如果 `Norm_Exposure` 极差，大幅扣分。
   * 如果检测到**大占比的主体人脸**闭眼，大幅扣分；若仅为背景小人脸，轻微扣分。
   * 扣分后如果 `Total_Score < 阈值`，才标记为 `Rating = 1`。
2. **连拍去重逻辑（相对评分制与极端保护）**：
   * 将未被否决的图片按混合特征进行图聚类。
   * 对于每个连拍组（组内图片数 > 1）：
       * 选出 `Total_Score` 最高的一张，设为该组的“封面图 (Best Image)”。
       * **极端保护机制（UI 兜底）**：即使该组最优秀的“封面图”分数极低（比如都在抓拍暗光废片，得分为 2 星），**也不要强行拔高它的星级欺骗用户**。保持其 2 星评分，但前端 UI 依然基于 `group_id` 和 `is_cover` 将其他废片折叠在其后，避免 UI 折叠框破裂。
       * **组内次优保留**：如果某张备选照片的总分 > `组内最佳分数 * 0.95`，则正常给予其 `Rating = 3` 或 `4`。
     * 只有组内明显较差（分数低于 `best * 0.85`）的照片，才标记为连拍冗余，设为 `Rating = 2`。
3. **独立照片评级（分数校准与硬阈值）**：
   * 星级代表的是“绝对的质量承诺”，不能使用 Percentile（百分位排名）导致“矮子里拔高个”。
   * **Score Calibration (分数标准化)**：由于包含人脸的照片和不包含人脸的照片在计算总分时使用了不同的权重体系，在最终映射星级前，必须对分数进行校准，确保 `Total_Score` 的绝对物理意义一致。
     * **校准实现约束**：校准函数必须满足 1) **单调递增**（不改变原有排序），2) **输出范围固定**（始终为 `[0,1]`），3) **参数固定**（硬编码或配置文件，绝对不可随当前目录的动态分布学习漂移）。
   * **绝对星级映射**（示例）：
     * `Calibrated_Score > 0.85` -> `Rating = 5` (极佳)
     * `Calibrated_Score > 0.70` -> `Rating = 4` (可用)
     * `Calibrated_Score > 0.50` -> `Rating = 3` (一般)
     * `Calibrated_Score <= 0.50` -> `Rating = 2` (较差)

### 4.3 技术失败降级与并发限制 (Fallback & Concurrency)
1. **模型崩溃 Fallback**：在任何步骤发生模型崩溃（如 ONNX Runtime 崩溃、显存耗尽导致 CLIP 失败），绝不应导致系统崩溃。必须捕获异常，**回退 (Fallback) 到只依赖客观物理指标（清晰度 + 曝光）打分**，并打上 `{"reason": "AI分析降级"}` 标签。
2. **重型推理并发限制**：在使用 `rayon` 或 `tokio` 遍历上千张照片时，必须对 CLIP/ONNX 推理节点施加严格的**并发信号量（如 Semaphore = 2~4）**。如果不限制并发，瞬间压入 16 个 GPU 推理任务会导致显存爆炸或排队死锁，性能反而剧烈下降。

### 4.4 可解释标签系统 (Explainability)
不仅要输出 `Rating` 星级，必须同时写入详细的评判原因。
```json
{
  "rating": 2,
  "reasons": ["严重失焦", "连拍重复"],
  "score_breakdown": {
    "blur": 0.12,
    "exposure": 0.90,
    "face": 0.0
  }
}
```
这能在前端 UI 上直接展示（如给照片打上“模糊”的红色角标），极大提升用户的信任度。

## 5. 用户可控层与缓存策略（产品级要求）

### 5.1 用户 Profile 偏好选择
自动选图不能是纯黑盒的。在启动选图任务时，应允许用户选择当前的“拍摄场景 (Profile)”，系统将根据 Profile 动态切换 `Total_Score` 中的各项权重：
*   **人像优先 (Portrait)**：大幅提升 `Face_Score` 权重，降低 `Blur_Score` 的容忍度（必须锐利），并开启严格的闭眼检测。
*   **风光静物 (Landscape)**：将 `Face_Score` 权重降为 0，大幅提升 `Aesthetic_Score` (构图美学) 的权重。
*   **活动抓拍 (Event)**：调低连拍去重的相似度阈值（允许保留更多动作变化），更看重“是否有笑容/互动”。

### 5.2 工程缓存与增量更新 (Cache & Incremental)
为防止反复分析同一批照片拖垮系统性能，必须将重计算提取出的 `Norm_Blur`、`CLIP_Embedding`、`Face_Stats` 等中间特征缓存起来。
*   **Cache Key 设计**：使用 `文件哈希 + 文件大小 + EXIF修改时间 + 预处理版本号 (preprocess_v1)` 作为缓存键，存入对应的 `.rrdata` 文件。预处理版本号能确保后续升级缩略图尺寸或模型时，旧缓存自动失效。
*   **增量更新**：如果用户往已分析的文件夹中新增了 100 张照片并重新点击“智能选图”，系统只需分析这 100 张，并在内存中与已有缓存记录合并后重新执行一次“聚类与打分规则”即可，实现秒级热启动。

## 6. 工程实现步骤 (MVP 阶段规划)

为保证项目可控且能快速验证价值，建议分阶段实现：

### Phase 1: 基础设施与基础评分 (MVP)
**目标**：跑通异步处理流，实现清晰度、曝光的打分，并完成基于时间窗的连拍去重。这是验证用户价值的第一步。
1. **扩展数据结构**：在 [image_processing.rs](file:///Users/qiaopeng/Desktop/owner/QRaw/src-tauri/src/image_processing.rs) 的 `ImageMetadata` 旁边定义 `AutoRatingStats` 结构体暂存特征。
2. **实现 RAW+JPEG 资产绑定**：在目录扫描阶段，将同名但不同后缀的文件（如 `IMG_123.CR3` 与 `IMG_123.JPG`）视为单一实体，避免互相竞争。
3. **实现 Laplacian 算子**：在 `image_processing.rs` 中手写一个基于 `rayon` 的拉普拉斯方差计算函数。

## 7. 评测指标定义（Evaluation Metrics）

在任何评分引擎落地前，必须构建离线评测闭环，以防止靠“感觉”调参导致系统性能劣化。必须构建涵盖不同场景（婚礼、人像、活动、风光）的黄金数据集。

### 7.1 核心指标
*   **Top-K 命中率 (Top-K Hit Rate)**：
    *   定义：算法给出的 Top K（如 5 星）照片中，与专业修图师手动选出的保留照片的重合度。
    *   目标：验证系统选出好照片的能力。
*   **废片召回率 (Trash Recall)**：
    *   定义：数据集中标注为绝对废片（失焦、闭眼、严重过曝）的照片，被算法成功打分为 1 星或标记剔除的比例。
    *   目标：验证系统的“兜底避坑”能力。
*   **用户改选率 (User Override Rate)**：
    *   定义：线上灰度测试中，用户对 AI 预选出的 4/5 星照片进行手动降级，或从隐藏的连拍堆里捞出 2 星照片的比例。
    *   目标：验证算法策略与真实用户审美的契合度。

## 8. 技术栈与性能考量

本方案**完全不使用 Python**，严格遵循 QRaw 的高性能本地化架构。

*   **并行计算**：严重依赖 `rayon`（CPU 密集型如 Laplacian）和 `tokio`（IO 密集型如读取文件和元数据）。
*   **内存管理**：避免将几千张图片的完整像素读入内存。所有评分（除了去重矩阵）均在缩略图（如 1024x1024 甚至 512x512）上即时计算完毕并丢弃图像数据，只保留轻量的 `AutoRatingStats` 结构体。
*   **ONNX 优化**：复用 `ort` 引擎，确保 CLIP 和未来的人脸模型能够利用 CoreML (macOS)、DirectML (Windows) 或 TensorRT 加速。
