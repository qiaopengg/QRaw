> 版本: 0.1
> 最后更新: 2026-05-12
> 状态: 需求澄清中，尚未进入实现
> 约束: 必须遵守 `AGENTS.md` 和 `docs/feature-integration-guidelines.md`

## 记录规则

本文档用于记录 `智能选图` 二开功能的连续需求讨论，避免后续遗漏上下文。

后续每一轮 Q&A 的结论都必须追加到本文档。未达成一致的问题必须进入 `待确认问题`，已经达成一致的问题必须进入 `已确认产品约束` 或对应技术章节。

## 功能背景

摄影师一次拍摄可能产生数百到上千张照片。逐张打开审查会消耗大量时间，因此需要一个完全离线、本地推理的智能辅助选图功能，帮助摄影师在摄影工作流早期完成初筛。

本功能目标不是删除照片，而是辅助用户识别好图、废片、重复图和待确认照片，并将结果以星级、精选/淘汰/待确认、颜色标签、分组推荐和简单解释的形式应用到图库。

## 已确认产品约束

### 输出维度

智能选图不仅输出星级，还必须输出以下辅助维度：

- 星级: 1 到 5 星，0 星视为未处理或未打星
- 精选
- 淘汰建议
- 待确认
- 颜色标签
- 相似/重复分组推荐
- 每张照片的简单解释

图库必须支持用户按这些维度查看和筛选图片。

### 不删除照片

智能选图只做打星、标记、分组和推荐，不执行删除。淘汰仅表示系统建议，不应触发文件删除。

### 不覆盖人工评分

不允许覆盖摄影师自己打的星级。

评分必须区分来源：

- `manual`: 摄影师手动评分
- `smart_culling`: 系统智能选图评分

如果现有 `.rrdata` 已有星级但没有来源字段，默认视为摄影师手动评分。系统只能修改自己写入的智能评分，不能覆盖人工评分。

### 打星语义

智能选图处于摄影流程的起始阶段，用于区分好图到废图的质量区间。只有用户筛选出的好图才进入后续修图流程。

当前星级与筛选状态的默认联动为：

- 5 星 / 4 星: 精选
- 3 星 / 2 星: 待确认
- 1 星: 淘汰建议
- 0 星: 未处理或无结论

### 颜色标签语义

颜色标签只作为用户自定义标识，不赋予系统固定含义。系统不得默认把某种颜色绑定为精选、淘汰或待确认。

V1 默认不由系统自动推荐颜色标签。复核页只提供手动或批量设置颜色标签能力，避免系统替用户定义颜色含义。

后续如需要自动颜色标签，只能通过用户显式创建的预设规则启用，例如用户自行定义 `精选 -> 绿色`。该规则必须保存在智能选图预设中，不能作为系统默认行为。

### 选图模式

启动智能选图前，应先让用户选择选图模式。不同模式影响评分权重和高优先级判断项。

V1 内置模式采用当前建议：

- 人像
- 婚礼/活动
- 儿童/家庭
- 风光
- 街拍/纪实
- 体育/动物/飞鸟
- 产品/静物
- 建筑/空间
- 通用模式

### 模式权重示例

不同模式应有不同的质量判断优先级：

- 人像模式: 人脸、眼睛、主体清晰度、人像表情权重高
- 婚礼/活动模式: 人像表情、多人合照最佳表情、主体清晰度、重复组最佳图权重高
- 风光模式: 整体清晰度、构图、美学、曝光和画面稳定性权重高，人脸权重低
- 产品/静物模式: 主体清晰度、曝光、色彩、构图、重复图差异权重高
- 体育/动物/飞鸟模式: 主体清晰度、运动瞬间、重复组最佳图权重高

### 人像表情判定

人像表情判定需要一个总开关，由用户决定是否启用。

启用后，用户可以组合启用并配置权重：

- 闭眼
- 糊脸
- 表情异常
- 微笑
- 多人合照最佳表情
- 主体是否看镜头

默认策略:

- 人像、婚礼/活动、儿童/家庭模式下默认开启
- 风光、产品/静物、建筑/空间模式下默认关闭

如果用户选择人像模式但照片中未检测到人脸，应降级为通用画质 + 美学评分，并在原因中写明 `未检测到人脸`。

多人合照中，如果部分人闭眼但主体表情好，不应直接降星，而应作为扣分项。只有闭眼人数比例过高或主体闭眼时，才显著影响星级。

### 相似/重复照片处理

相似或重复照片应聚合成组。

默认展示:

- 只显示系统判定的最优图
- 其他相似照片折叠
- UI 显示组内数量，例如 `+7 相似`

用户必须可以：

- 展开查看组内全部图片
- 手动指定组内最优图
- 配置每组保留前 N 张

### 清晰度判定

清晰度判定对象随选图模式变化：

- 人像优先判断人脸、眼睛和主体清晰度
- 风光优先判断整体画面和关键区域清晰度
- 产品/静物优先判断主体清晰度
- 体育/动物/飞鸟优先判断主体清晰度和运动瞬间可用性

### 美学评分

美学评分支持两类策略：

- 通用模型评分
- 用户自定义偏好评分

用户自定义偏好需要支持：

- 简单滑块配置
- 保存为个人选图风格预设

已确认需要纳入偏好的因素包括：

- 暗调
- 胶片感
- 浅景深
- 抓拍情绪
- 色彩鲜明
- 构图稳定

这些偏好会影响综合评分。

### 智能选图预设

智能选图配置需要支持保存为预设。

预设内容至少包含：

- 选图模式
- 人像表情总开关
- 人像表情子项开关与权重
- 综合评分权重
- 用户偏好滑块
- 个人选图风格预设
- 每组保留前 N 张

V1 默认系统预设：

- 严格筛选
- 均衡筛选
- 宽松筛选

用户自定义预设保存为独立文件，避免污染主设置：

```text
smart-culling-presets.json
```

建议位置：

```text
<app_data_dir>/smart-culling-presets.json
```

### 离线推理

智能选图必须完全离线、本地运行，不依赖云端服务。

模型交付方式:

- V1 复用上游 RapidRAW AI Tagging 的 CLIP ONNX 模型，支持在智能选图弹窗内触发下载
- 下载模型仅用于本地缓存和本地推理，不上传图片或元数据
- 人脸/表情、专用美学 head 等后续模型仍需按 license 审查后再决定是否自动下载
- 应支持模型缺失时的清晰提示和目录打开引导

模型体积约束:

- 本地模型总量 2GB 内可接受

性能目标:

- 1000 张 RAW 在中端设备及以上尽量控制在 2 分钟内
- 允许使用硬件加速
- 应尽可能利用并行解码、批处理推理、GPU/ANE/NPU/Metal/CoreML/ONNX Runtime 等能力

### RAW 分析策略

用户要求分析来源使用 RAW 原图。

技术建议:

- 文件来源必须是 RAW
- 但模型不应对全分辨率 RAW 逐项做完整推理
- 应从 RAW 解码得到分析用降采样图，例如长边 1024/1536/2048
- 对清晰度、重复度、人脸和美学评分采用分层推理:
  - 快速阶段: 内嵌预览或快速 RAW 解码图，用于粗筛、相似分组和任务调度
  - 精判阶段: 对候选图、重复组最佳图和待确认图使用更高分辨率分析
  - 人像局部阶段: 对脸部、眼睛等局部区域单独裁剪分析

该策略在时间和质量之间更平衡，也更接近 1000 张 RAW / 2 分钟的性能目标。

### 结果解释

解释信息采用简单解释即可，例如：

- 清晰度高
- 人脸清晰
- 表情正常
- 与 8 张相似图中评分最高
- 美学评分较高
- 曝光或构图存在风险

不要求 V1 提供复杂的逐层权重贡献图。

### 人工复核

需要人工复核流程。

智能选图结果应用前，应允许用户查看、确认、调整或撤销。结果不能直接造成不可逆操作。

### 入口位置

在左侧新增 `智能选图` 入口，与社区预设等入口并列。

点击入口后弹出配置弹窗。用户完成配置后开始选图分析。分析完成后，将智能选图结果应用到图片，并在现有 Library 模块中显示。

原项目已有手动打星能力，智能选图应复用现有图库评分和筛选显示能力。

### 任务范围与执行流程

默认分析范围为当前文件夹全部图片。

配置弹窗必须支持用户选择分析范围。V1 建议支持：

- 当前文件夹全部图片
- 当前选中图片
- 当前筛选结果

V1 默认不递归子文件夹，避免任务范围失控。若后续需要递归，应作为显式选项。

已有人工评分的照片应跳过，不进入智能选图处理。跳过时必须说明原因，例如 `已存在人工评分，智能选图不会覆盖`。

已有用户修图调整的照片默认跳过，避免把已经进入后期流程的照片重新纳入初筛。跳过时必须说明原因，例如 `已存在修图调整，智能选图默认跳过`。

配置弹窗可提供 `包含已修图照片` 的显式选项，默认关闭。实现时必须只识别用户可见的非默认调整，不能把空调整对象或默认参数误判为已修图。

智能选图完成后，必须先进入用户确认流程。未经用户确认，不应直接应用评分和状态。

分析完成后应自动打开复核页面，并在顶部显示任务摘要。

复核页面至少支持：

- 接受全部
- 拒绝全部
- 按组接受
- 修改星级
- 修改组内最佳
- 只应用精选
- 只应用淘汰建议

复核页需要提供相似组对比视图，展示组内缩略图、系统分数和原因，帮助用户选择最佳图。

淘汰建议必须明确提示 `不会删除文件`，避免用户误解。

复核页需要支持按系统原因筛选，例如：

- 闭眼
- 糊脸
- 重复
- 曝光异常
- 主体不清晰
- 未检测到人脸
- 可信度降低

这些原因筛选默认只影响复核页面查看，不应自动写入普通用户标签。应用结果后，可将原因代码保存在 `featureData.smartCulling.reasonCodes` 中，用于历史报告和智能筛选。

### RAW + JPEG 处理

RAW + JPEG 同名照片视为同一拍摄资产，不做重复打星。

默认策略：

- 同名 RAW 存在时，以 RAW 为主资产参与分析与评分
- JPEG 作为同组附属文件，不重复生成智能评分
- 评分写入主资产对应的 `.rrdata`
- Library 展示时应避免 RAW/JPEG 同名资产出现重复智能结果

### 虚拟副本处理

项目中虚拟副本是指带有 `?vc=` 的路径，它代表同一原始照片的一份编辑副本，而不是另一张真实拍摄照片。

暂定策略：

- V1 智能选图默认以原始拍摄文件为单位分析
- 虚拟副本不单独参与智能选图，避免同一张照片重复评分
- 如果用户当前视图包含虚拟副本，复核界面应说明其智能结果来源于原始文件
- 虚拟副本上的人工评分仍按人工评分保护规则处理

该策略已在第 5 轮确认。

### Library 筛选

Library 需要支持智能选图相关筛选：

- `智能精选`
- `智能待确认`
- `智能淘汰建议`
- `有智能建议`
- `未经智能选图处理`

其中 `未经智能选图处理` 用于避免重复处理已经分析过的图片。

筛选项显示名称使用中文。

Library 缩略图展示必须遵守人工优先：

- 如果照片存在人工星级，主星级只显示人工星级
- 智能建议以次级徽标展示，例如 `AI 4` 或 `智能建议 4 星`
- 如果照片没有人工星级，且用户确认应用了智能评分，可以在主星级位置显示该评分，但必须带有智能来源标识
- 鼠标悬停或详情面板中应展示评分来源、智能状态、主要原因和任务时间

### 重复组范围

相似/重复分组仅在本次智能选图任务范围内生效。

V1 不做跨图库历史分组，避免索引成本、隐私边界和历史结果一致性问题。

### 重复组保留数量

`每组保留前 N 张` 指的是：当一组连拍或高度相似照片出现时，系统默认推荐保留该组评分最高的前 N 张作为候选，其余折叠为重复项。

暂定策略：

- 默认每组保留 1 张
- 用户可以在配置弹窗中调整 N
- 用户可以在复核页面手动修改组内最佳图

该策略已在第 5 轮确认。

### 评分阈值

星级映射阈值采用系统内置固定策略，V1 不向用户暴露阈值调节。

用户偏好通过选图模式、权重预设和个人风格预设影响综合评分，不直接编辑星级阈值。

### 进度展示

智能选图必须显示详细进度粒度。

建议阶段：

- 准备任务
- RAW 解码 / 生成分析图
- 清晰度分析
- 相似度分析与分组
- 人脸与表情分析
- 美学评分
- 综合评分
- 生成复核结果
- 写入 `.rrdata`

### 取消策略

用户取消任务时，本次任务结果全部丢弃。

不得部分写入 `.rrdata`。如后续为了性能需要分批写入，也必须实现事务式或可回滚策略，确保取消后不会留下半成品智能结果。

用户主动取消不生成正式任务报告，只保留必要的运行状态提示。

### 预览模式

V1 需要支持 `仅分析不写入` 模式，用于用户先查看系统判断。

预览模式规则：

- 不写入照片 `.rrdata`
- 不改变 Library 主星级、智能状态或颜色标签
- 分析完成后仍打开复核页，但复核页处于只读预览状态
- 用户可在复核页选择转为正常应用流程，再进行确认写入
- 可导出预览 PDF 报告，报告中必须标记 `预览结果，未写入照片`

### 模型缺失与降级策略

允许在部分非关键模型缺失或启动失败时降级运行，但必须提示用户。

提示应说明：

- 哪个模型或能力不可用
- 本次分析会缺少哪些判断项
- 结果可信度可能受到什么影响

关键模型缺失时应禁止开始任务。V1 需要在模型清单中区分 `required` 与 `optional`。

降级运行时，结果必须标记 `可信度降低`，并在复核页和任务报告中说明缺失能力。

### 失败报告

智能选图任务异常失败时，需要生成失败报告，帮助用户定位问题并支持后续排障。

失败报告至少包含：

- 任务开始时间和失败时间
- 分析范围
- 已分析数量
- 失败图片数量
- 失败图片名称或路径
- 失败原因
- 模型加载状态
- 是否发生降级运行

任务整体失败时不得写入照片 `.rrdata`。如果只是部分图片失败，应在复核页和最终 PDF 报告中列出失败图片，其余成功结果仍可由用户确认应用。

### 任务报告

智能选图复核完成并应用结果后，需要生成本次任务报告。

报告至少包含：

- 分析总数
- 跳过数量
- 精选数量
- 待确认数量
- 淘汰建议数量
- 标注时间
- 图片名称
- 打星等级
- 打星原因
- 跳过原因

示例摘要：

```text
分析 860 张，跳过 40 张，精选 120 张，待确认 300 张，淘汰建议 400 张
```

人工评分照片被跳过时，需要在报告中单独列出。

任务报告导出格式固定为 PDF。

报告持久化策略：

- `.rrdata` 中保留每张图片的智能选图摘要和任务 ID
- App 数据目录保留任务级报告数据，用于历史查看、撤销和重新导出 PDF
- 用户可从智能选图历史任务中导出 PDF 报告
- 最近 10 次历史报告均可重新打开查看

历史任务保留最近 10 次。超过 10 次时，自动清理最旧任务的任务级报告数据，但不删除已经写入图片 `.rrdata` 的智能选图结果。

如果某次智能选图结果被撤销，历史报告仍应保留，但任务状态必须标记为 `已撤销`。重新打开或重新导出 PDF 时，也必须展示 `已撤销` 状态，避免历史记录与当前照片状态混淆。

PDF 缩略图策略：

- 摘要页不展示缩略图，避免 PDF 过大
- 明细页可选展示小缩略图
- 如果缩略图生成失败，报告仍应导出，只记录图片名称和路径

### 一键撤销

智能选图结果应用后，必须支持“一键撤销上次应用”。

撤销要求：

- 只撤销最近一次智能选图任务写入的结果
- 不影响用户在任务后手动修改的评分或标签
- 不影响智能选图之前已经存在的人工评分
- 撤销所需的快照或变更日志应写入 `.rrdata` 或任务级报告数据中

撤销入口由产品设计确定为：

- 智能选图入口弹窗中的最近任务列表
- 最近一次任务完成后的 toast 操作按钮
- 任务报告详情页中的撤销按钮

撤销按钮必须清晰标识只撤销智能选图写入内容，不撤销用户手动操作。

### 复核排序

复核页面默认排序：

1. 精选
2. 待确认
3. 淘汰建议

重复组内部按系统综合分从高到低排序。

复核页中用户修改星级后，该评分视为用户确认后的智能结果，不再被本次任务自动覆盖。任务完成后用户再次手动修改，则视为人工评分，后续智能选图不得覆盖。

### 后台运行

智能选图应支持后台运行。

用户可以在分析期间继续浏览图库，但限制如下：

- 同一时间只允许一个智能选图任务运行
- 任务运行中不能启动第二个智能选图任务
- 任务进度应可在入口或全局任务状态中查看
- 用户可以取消当前任务
- 取消后本次任务结果全部丢弃
- 用户切换文件夹时，任务继续运行
- 结果应用前仍必须回到复核页确认

任务完成后应发送系统通知，不播放声音。

### 样片评测集

用户有真实样片集，可以作为后续评测标准。

建议建立评测记录，覆盖：

- 样片规模
- 人工金标准
- 系统打星结果
- 误判样例
- 各模式评分效果
- 每次模型或权重调整后的对比

## `.rrdata` 持久化源码确认

已查看当前源码，`.rrdata` 相关事实如下：

- `parse_virtual_path()` 会将原图映射到同目录 sidecar:
  - 原图: `<filename>.rrdata`
  - 虚拟副本: `<filename>.<vcId>.rrdata`
- 当前 `ImageMetadata` 定义位于 `src-tauri/src/image_processing.rs`
- 当前字段:

```rust
pub struct ImageMetadata {
    pub version: u32,
    pub rating: u8,
    pub adjustments: Value,
    #[serde(default)]
    pub tags: Option<Vec<String>>,
}
```

- 当前评分写入函数为 `set_rating_for_paths()`，直接修改 `metadata.rating`
- 当前颜色标签通过 `tags` 保存，颜色 tag 前缀来自 `COLOR_TAG_PREFIX`
- 当前 XMP 同步开启时，手动评分和颜色标签可能同步到 XMP

### 智能选图持久化初步约束

智能选图必须写入对应照片的 `.rrdata`，但不能破坏现有字段语义。

需要记录的信息至少包括：

- 智能评分
- 评分来源
- 精选/淘汰/待确认状态
- 简单解释
- 相似组 ID
- 是否为组内系统最优
- 用户是否手动改写过系统最优
- 使用的选图模式
- 使用的模型版本
- 使用的权重预设版本
- 分析时间

由于当前 `ImageMetadata` 没有来源字段，后续实现时需要设计兼容扩展。初步建议优先考虑一次性增加通用扩展字段，例如：

```json
{
  "version": 1,
  "rating": 4,
  "adjustments": {},
  "tags": ["color:red"],
  "featureData": {
    "smartCulling": {
      "rating": 4,
      "ratingSource": "smart_culling",
      "status": "selected",
      "explanations": ["人脸清晰", "与 8 张相似图中评分最高"],
      "groupId": "smart-group-...",
      "isGroupBest": true,
      "mode": "portrait",
      "modelVersion": "...",
      "presetVersion": "..."
    }
  }
}
```

注意：不建议把智能选图结构直接塞入 `adjustments`，因为现有图库逻辑会根据 `adjustments` 判断照片是否已编辑，可能导致未修图照片被误标记为已编辑。

## 架构决策

以下决策由当前项目基建推导得出，用于替代早期 `待确认问题`。除非后续实现中发现硬性技术阻塞，否则按本章节执行。

### 决策 1: `.rrdata` 扩展 schema

采用通用扩展字段 `featureData.smartCulling`。

原因：

- 当前 `ImageMetadata` 只有 `version/rating/adjustments/tags`
- `adjustments` 已被现有图库逻辑用于判断照片是否已编辑
- 将智能选图数据写入 `adjustments` 会导致未修图照片被误标记为已编辑
- `tags` 当前承担颜色标签和普通 tag，不适合承载复杂结构
- `featureData` 可以作为所有二开 feature 的通用持久化扩展点，后续也可复用

实现时应将 `ImageMetadata` 扩展为向后兼容结构：

```rust
#[derive(Serialize, Deserialize, Debug, Clone)]
pub struct ImageMetadata {
    pub version: u32,
    pub rating: u8,
    pub adjustments: Value,
    #[serde(default)]
    pub tags: Option<Vec<String>>,
    #[serde(default, rename = "featureData", skip_serializing_if = "Option::is_none")]
    pub feature_data: Option<Value>,
}
```

### 决策 2: 智能评分写入策略

智能选图结果需要同时写入：

- `featureData.smartCulling.rating`: 永久记录系统评分
- 顶层 `rating`: 仅当顶层评分可安全视为系统可接管时才写入

安全接管规则：

- 如果 `.rrdata` 无评分或顶层 `rating == 0`，可以写入智能评分到顶层 `rating`
- 如果 `.rrdata` 已有评分且没有 `featureData.smartCulling.ratingAppliedToTopLevel == true`，默认视为人工评分，不覆盖顶层 `rating`
- 如果顶层评分上一次由智能选图写入，可由新一次智能选图更新
- 用户手动改星后，应将该照片视为人工评分接管，后续智能选图不得覆盖顶层 `rating`

这样可以复用现有 Library 的评分显示、排序和筛选能力，同时保护用户手动评分。

### 决策 3: 人工评分存在时的智能评分处理

人工评分存在时，系统仍应记录智能评分，但不作为主评分显示。

规则：

- 顶层 `rating` 保持人工评分
- `featureData.smartCulling.rating` 保存系统建议评分
- `featureData.smartCulling.status` 保存系统建议状态
- UI 可在智能选图复核面板中展示“系统建议”，但 Library 主评分仍以人工评分为准

### 决策 4: XMP 同步策略

智能选图结果默认严格只写 `.rrdata`，不主动同步到 XMP。

原因：

- 当前设置页说明 XMP 同步面向 ratings、color labels、tags 与外部编辑器兼容
- 智能选图是辅助初筛结果，属于 QRaw 内部决策数据
- 将系统建议同步到 XMP 可能污染摄影师在 Lightroom、Capture One 等外部软件中的人工评级

例外规则：

- 如果顶层 `rating` 被安全写入为智能评分，而用户开启了现有 XMP 同步，当前基础设施可能会同步顶层 rating
- V1 不新增智能选图专用 XMP 字段
- 若未来需要导出系统建议，应作为显式导出功能，而不是默认同步

### 决策 5: 本地模型目录和清单

V1 模型分两层管理：

- 共享 CLIP 模型复用上游 RapidRAW 模型缓存，可由智能选图弹窗触发下载。
- 智能选图专用模型仍放在 feature 专属目录，并通过 manifest 描述，避免与上游模型分叉。

基于 Tauri 当前路径使用习惯，默认目录建议为：

```text
<app_data_dir>/models/
<app_data_dir>/models/smart-culling/
```

模型目录内必须包含模型清单：

```text
manifest.json
```

清单记录：

- 模型包版本
- 每个模型文件名
- 模型用途，例如 aesthetics、face、embedding、expression、sharpness
- 输入尺寸
- 推理后端要求
- 校验信息

缺失模型时：

- 智能选图入口仍可显示
- 配置弹窗提示模型缺失
- 提供“打开模型目录”操作
- 提供“下载模型”操作下载共享 CLIP 模型
- 允许降级基础分析，结果必须标记可信度降低

### 决策 6: 硬件加速优先级

V1 推理后端优先级：

1. 平台原生硬件加速
   - macOS 优先 CoreML / Metal
   - Windows 优先 ONNX Runtime DirectML 或 CUDA（如可用）
   - Linux 优先 ONNX Runtime CUDA / Vulkan / CPU fallback
2. ONNX Runtime 通用后端
3. CPU fallback

工程原则：

- 后端选择逻辑必须位于 `src-tauri/src/features/smart_culling/`
- 不把推理后端判断散落到上游 UI 或核心模块
- 必须支持能力探测和失败回退
- 1000 张 RAW / 2 分钟目标依赖硬件，但实现必须尽量通过批处理、并行、缓存和分层推理逼近

### 决策 7: 样片评测集

真实样片集不直接提交到仓库。

仓库只保留评测协议和可选金标准格式：

```text
docs/smart-culling-evaluation.md
fixtures/smart-culling/.gitkeep
```

真实 RAW 样片放在本地或外部存储，通过配置路径引用。

原因：

- RAW 样片体积大，不适合进入 Git 仓库
- 真实照片可能包含隐私或商业内容
- 保留评测协议即可复现实验流程

建议金标准格式：

```json
{
  "dataset": "portrait-session-001",
  "items": [
    {
      "path": "relative/or/external/path.ARW",
      "manualRating": 5,
      "expectedStatus": "selected",
      "groupId": "group-001",
      "isGroupBest": true,
      "notes": ["sharp eyes", "best expression"]
    }
  ]
}
```

## Q&A 记录

### 第 1 轮: 背景与初始共识

用户确认：

1. 输出不仅是星级，还需要精选、淘汰、待确认、颜色标签、分组推荐，并支持按这些维度查看图片。
2. 不允许覆盖摄影师自己打星，需要记录评分来源，只能修改系统打星。
3. 只做打星和推荐，不做删除。
4. 需要支持多种拍摄类型；启动时先让用户选择选图模式，并根据模式调整高优先级评分策略。
5. 人像表情判定需要总开关；启用后可组合闭眼、糊脸、表情异常、微笑、多人合照最佳表情、主体是否看镜头等权重。
6. 相似重复照片只显示最优，其他折叠，但支持用户手动判定最优。
7. 清晰度判断随选图模式变化，例如人像优先人像清晰，风光优先风光整体质量。
8. 美学评分同时支持通用模型和用户自定义偏好；暗调、胶片感、浅景深、抓拍情绪等偏好会影响评分。
9. 必须完全离线本地推理。
10. 本地模型体积 2GB 可接受；1000 张 RAW 希望 2 分钟内完成。
11. 使用 RAW 原图作为分析来源。
12. 入口放在项目社区预设左侧区域。
13. 需要结果解释。
14. 需要人工复核流程。
15. 评分需要持久化，写入对应照片 `.rrdata` 配置文件。
16. 智能选图作为第二个二开功能，必须独立维护。
17. 第一版希望整体一起做，因为打星是综合判定结果，各维度缺一不可。
18. 用户有真实样片集。

### 第 2 轮: 模式、星级、持久化和入口确认

用户确认：

1. V1 内置选图模式采用建议列表：人像、婚礼/活动、儿童/家庭、风光、街拍/纪实、体育/动物/飞鸟、产品/静物、建筑/空间、通用模式。
2. 打星用于摄影流程早期初筛，核心是区分好图到废图的区间；好图进入后续修图流程，废图通常不进入下一流程。
3. 星级联动状态采用：5/4=精选，3/2=待确认，1=淘汰建议。
4. 颜色标签只做用户自定义标识，没有系统固定含义。
5. 已有星级但没有来源字段时，默认视为摄影师手动评分。
6. `.rrdata` 结构需查看源码确认；本轮已完成初步源码确认，记录见 `.rrdata 持久化源码确认`。
7. 1000 张 RAW / 2 分钟目标必须尽量满足，面向市面中端设备以上，允许硬件加速。
8. RAW 分析策略采用时间与质量平衡方案：以 RAW 为来源，内部生成降采样分析图，并采用快速阶段、精判阶段和局部阶段的分层推理。
9. 模型由用户手动放置到模型目录。
10. 结果解释采用简单解释即可。
11. 人像开关默认策略合理：人像/婚礼/儿童默认开启，风光/产品/建筑默认关闭。
12. 重复组需要支持每组保留前 N 张。
13. 用户自定义偏好既需要简单滑块，也需要保存为个人选图风格预设。
14. 左侧新增智能选图入口；点击后出现配置弹窗，配置完成后开始选图，最终将打星应用到图片，结果在 Library 模块显示。
15. 真实样片集可以作为自动化评测集。

### 第 3 轮: 待确认问题转为架构决策

用户要求由 Codex 阅读项目基建架构后进行最优判断，不再由用户逐条确认。

基于现有代码得出以下判断：

1. `.rrdata` 使用 `featureData.smartCulling` 作为智能选图持久化扩展点。
2. 智能评分始终记录在 `featureData.smartCulling.rating`；只有无人工评分或顶层评分由系统写入时，才同步到顶层 `rating`。
3. 人工评分存在时，智能评分只作为系统建议保留，不作为 Library 主评分显示。
4. 智能选图结果默认只写 `.rrdata`，不新增专用 XMP 同步；顶层 rating 若被安全写入，则可能沿用现有 XMP 同步机制。
5. 模型目录采用 `<app_data_dir>/models/smart-culling/`，通过 `manifest.json` 描述模型包。
6. 硬件加速优先级为平台原生加速、ONNX Runtime 通用后端、CPU fallback。
7. 真实样片集不入库，仓库只保留评测协议和金标准格式。

详细说明见 `架构决策` 章节。

### 第 4 轮: 任务范围、复核流程与降级策略

用户确认：

1. 默认作用范围为当前文件夹。
2. 默认分析当前文件夹全部图片，但支持用户选择分析范围。
3. 已有人工评分的照片跳过，不做处理，但必须说明原因。
4. RAW + JPEG 同名照片视为一体，不做重复打星。
5. 用户暂不理解虚拟副本概念，需后续解释。
6. 智能选图完成后必须由用户确认。
7. 复核页面按建议支持：接受全部、拒绝全部、按组接受、修改星级、修改组内最佳、只应用精选、只应用淘汰建议。
8. 需要智能建议相关独立筛选项。
9. 需要“只看未处理照片”能力。
10. 重复组只在本次任务范围内分组。
11. 用户暂不理解“每组保留前 N 张”，需后续解释。
12. 评分阈值固定，不提供用户调整。
13. 分析过程显示详细进度粒度。
14. 用户取消任务时，本次结果全部丢弃。
15. 允许部分能力降级运行，但需要提示。

基于现有架构补充的默认建议：

- V1 范围选择支持当前文件夹、当前选中图片、当前筛选结果。
- V1 默认不递归子文件夹。
- 虚拟副本默认不单独参与智能选图，智能结果跟随原始文件。
- 每个重复组默认保留 1 张候选，可配置 N，并可在复核页手动调整。
- 模型清单需区分 required 与 optional，关键模型缺失则禁止开始，非关键模型缺失可降级运行。

第 5 轮已确认虚拟副本和每组保留前 N 张策略。

### 第 5 轮: 虚拟副本、重复组、报告、撤销和后台运行

用户确认：

1. 虚拟副本按建议处理：智能选图只分析原始文件，虚拟副本不单独打星，避免同一张照片重复评分。
2. 每组保留前 N 张按建议处理：默认 N=1，用户可在配置弹窗调整。
3. 需要生成本次任务报告。报告需要体现分析总数、跳过数量、精选数量、待确认数量、淘汰建议数量，并记录标注时间、图片名称、打星等级和原因。
4. 跳过的人工评分照片需要在任务报告里单独列出。
5. 需要支持一键撤销上次应用。
6. 复核页面默认排序按建议执行：精选在前、待确认居中、淘汰建议在后；重复组内部按系统分数降序。
7. 智能选图支持后台运行；用户可以继续浏览图库，但不能同时启动第二个智能选图任务。

### 第 6 轮: PDF 报告、历史任务、撤销入口和中文筛选

用户确认：

1. 任务报告导出为 PDF。
2. 报告导出格式只需要 PDF。
3. 一键撤销入口由 Codex 设计。
4. 智能选图历史任务保留最近 10 次。
5. 智能选图筛选项显示名称使用中文。

基于现有产品交互补充的默认设计：

- `.rrdata` 中保留每张图片的智能选图摘要和任务 ID。
- App 数据目录保留任务级报告数据，用于历史查看、撤销和重新导出 PDF。
- 一键撤销入口放在智能选图入口弹窗的最近任务列表、任务完成 toast、任务报告详情页。

### 第 7 轮: 复核打开、后台任务、报告细节和通知

用户确认：

1. 分析完成后自动打开复核页面，并显示任务摘要。
2. 后台运行时，如果用户切换文件夹，任务继续运行；结果应用前仍回到复核页确认。
3. PDF 报告缩略图策略按建议执行：摘要页不放缩略图，明细页可选展示小缩略图。
4. 最近 10 次历史报告均可重新打开查看，并可重新导出 PDF。
5. 降级运行时，结果需要标记 `可信度降低`，并在报告和复核页显示。
6. 任务完成后只做系统通知，不做声音。
7. 复核页修改星级视为用户确认后的智能结果；后续手动修改则视为人工评分。

### 第 8 轮: 预设、人像降级、合照扣分和相似组对比

用户确认：

1. 智能选图配置需要保存为预设。
2. 默认提供系统预设：严格筛选、均衡筛选、宽松筛选。
3. 用户自定义预设保存为独立文件，不写入主设置。
4. 人像模式未检测到人脸时，降级为通用画质 + 美学评分，并说明 `未检测到人脸`。
5. 多人合照中部分人闭眼但主体表情好时，不直接降星，只作为扣分项；主体闭眼或闭眼比例过高时才显著影响星级。
6. 淘汰建议需要强提示 `不会删除文件`。
7. 复核页需要相似组对比视图，展示组内缩略图、分数和原因。

### 第 9 轮: 产品侧默认决策

用户要求产品侧由 Codex 基于架构思想支持决策。

决策如下：

1. 颜色标签 V1 默认不由系统自动推荐，只在复核页支持用户手动或批量设置；如未来需要自动颜色标签，必须由用户显式预设定义。
2. 已有用户修图调整的照片默认跳过，保护已经进入后期流程的图片；配置弹窗可提供 `包含已修图照片` 选项，默认关闭。
3. Library 缩略图展示遵守人工优先：人工星级作为主星级，智能建议作为次级徽标；无人工星级且用户确认应用智能评分时，可展示智能星级但必须标识来源。
4. 撤销智能选图结果后，历史报告继续保留，但任务状态标记为 `已撤销`，重新导出 PDF 时也展示该状态。
5. 任务异常失败时需要生成失败报告；整体失败不得写入 `.rrdata`，部分图片失败时在复核页和最终 PDF 中列出失败项。
6. 复核页需要支持按原因筛选，例如闭眼、糊脸、重复、曝光异常、主体不清晰、未检测到人脸、可信度降低。
7. V1 需要支持 `仅分析不写入` 预览模式；该模式不改照片 `.rrdata`，可打开只读复核页，也可转为正常应用流程。

### 第 10 轮: 技术架构与选型

用户要求基于项目基建架构和本文档需求，结合全网调研，设计可安全健康落地的最优技术架构和选型。

已新增技术架构文档：

```text
docs/smart-culling-technical-architecture.md
```

核心决策：

1. V1 采用 `Rust/Tauri 原生流水线 + ONNX Runtime 本地模型 + 独立 feature UI`，不采用 Python sidecar、云推理或侵入式改造。
2. 前端新增 `src/features/smart-culling/`，后端新增 `src-tauri/src/features/smart_culling/`。
3. 通过通用 Library feature slot 接入左侧/图库入口、复核页、报告页、缩略图徽标和智能筛选，不把智能选图业务写入上游组件。
4. V1 复用 `<app_data_dir>/models/clip_model.onnx` 与 `clip_tokenizer.json`，由智能选图弹窗触发下载；`<app_data_dir>/models/smart-culling/manifest.json` 记录该共享模型引用。
5. 推理运行时使用现有 `ort` / ONNX Runtime，按平台能力启用 CoreML、DirectML、CUDA，失败后 CPU fallback。
6. 相似分组采用 `image_hasher` 感知哈希快速分组 + CLIP embedding 二次确认。
7. 美学评分采用 CLIP/OpenCLIP image encoder + LAION-style aesthetic head；NIMA-style 小模型作为备选。
8. 人脸/表情采用可插拔 SCRFD/BlazeFace ONNX adapter，模型 license 未确认前不得默认打包。
9. `.rrdata` 必须增加通用 `featureData` 扩展字段，智能结果写入 `featureData.smartCulling`。
10. 任务写入采用事务式 `.rrdata` 更新，支持失败回滚、最近一次智能结果撤销和 PDF 报告。

### 第 12 轮: 模型下载与 CLIP 辅助推理

用户要求下载模型，并保证功能全流程无阻塞。

本轮实施范围：
