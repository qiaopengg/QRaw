# 执行摘要  
RapidRAW 核心采用 AGPL-3.0 许可证【89†L19-L22】。根据 AGPL-3.0，**只要**您“传输/分发”或提供修改后的软件给他人，就必须同时提供完整源代码【82†L123-L131】【61†L355-L363】。但是，如果仅限**内部使用**（不对外分发），则不触发开源义务【86†L1760-L1766】【82†L93-L100】。AGPL 的“网络互动”条款还规定：如果您将修改版软件部署为网络服务并允许用户远程使用，必须向这些用户免费提供源码【61†L355-L363】【84†L108-L113】。因此，在以下场景中遵循以下原则：**二次开发后凡是将 AGPL 代码或其衍生物与闭源模块结合并分发（编译进二进制、同包发布）都需开源**；若闭源模块完全运行在远端，与 AGPL 程序仅通过标准网络接口通信，可视为单独工作，可保持闭源；仅公司内部使用不需要开源；以 SaaS 形式提供服务时用户使用 AGPL 软件则需公开源码。报表中引用了 AGPL 官方条款和 FSF/GNU FAQ 等权威资料【82†L123-L131】【76†L1258-L1262】【86†L1760-L1766】。下文详细分析了“派生作品”判定、五种常见集成场景的合规性、不同许可混用风险和商业化可选策略，并给出文档留存与合规清单。最后给出结论：**如果将闭源风格迁移模块编译进或随 RapidRAW 一起分发，则必须开源；如果仅在后台远程提供服务可避免这一要求**。建议的行动包括咨询法律意见、设计清晰的插件接口、严格标注许可文件等。相关风险提示已在报告中指出。若需法律意见，请咨询专业律师。

## AGPL-3.0 触发条件与关键条款  
**分发义务：** AGPL-3.0 继承了 GPL-3.0 的强制开源要求，即凡**“传输(convey)”**带有本许可证软件或衍生版本的行为，都必须提供其完整源代码【82†L123-L131】。AGPL 将网络使用视为分发：若将修改版软件通过服务器提供给用户，必须向这些用户提供源码【61†L355-L363】【84†L108-L113】。例如，《AGPL-3.0》第13节明确：“当您修改本程序并通过网络向用户提供远程交互时，必须让这些用户能够免费获取您版本的相应源代码”【61†L355-L363】。AGPL 要求附带的程序界面显示许可证通知，并提供源码获取方式【82†L135-L143】【84†L108-L113】。

**内部使用例外：** 只在公司内部或个人环境中安装运行软件，不对外发布，不算传输【86†L1760-L1766】【82†L93-L100】。GPL/AGPL FAQ 指出：“在一个组织内复制多个副本并使用，不被视为对外分发”，因此不触发开源义务【86†L1760-L1766】。惟一例外是，当组织将软件副本转给其他组织或个人使用时，即视为分发。  

**聚合作品例外：** 如果 AGPL 代码与其他“独立、按原许可无衍生关系”的代码捆绑分发（汇编成一个安装包或分发介质），则可以不将许可证强加到后者；这种情况称为“聚合(aggregate)”【82†L139-L146】。即如果闭源模块真为独立程序并通过标准接口调用，不与 AGPL 核心紧密交互，可视作聚合而非衍生，这种情况下闭源部分可不受 AGPL 约束【76†L1258-L1262】。但如果联系密切（如动态链接、函数调用、共享数据结构），则应视为单一结合程序，全体必须遵守 AGPL【76†L1233-L1242】【76†L1258-L1262】。

## 派生作品/结合作品判定标准  
根据 FSF/GNU GPL FAQ，用何许可要求取决于“程序与插件是否构成单一作品”【76†L1258-L1262】。一般原则：  
- **单一结合**：若 RapidRAW 与闭源模块**紧密集成**（如在同一进程内动态链接、共享复杂数据结构或功能调用），则两者构成一个结合程序。根据 GPL 原则（亦适用于 AGPL），整个结合体都必须在 GPL/AGPL 许可下发布【76†L1258-L1262】。此时闭源模块等效衍生自 AGPL 代码，必须开源；否则违反许可证。  
- **分离程序**：若闭源模块完全**独立运行**，仅通过诸如文件、命令行、简单网络接口等机制被 RapidRAW 调用，则可视为分离的程序。这种情况下，闭源部分可维持其许可，不受 AGPL 约束【76†L1258-L1262】。例如，主程序仅以简单 fork/exec 调用插件，插件只返回计算结果，未共享内部数据，可不要求遵守 GPL。  
- **网络调用**：如果闭源功能通过网络 API 与 AGPL 程序交互，理论上可视为“分离”但存在争议【80†L161-L168】。业界观点分歧：一种观点认为因通过标准 API 通信可归类为分离系统；另一观点（FSF 风格）认为若最终应用依赖该功能，则整个服务或可视为衍生体【80†L165-L172】。因此，此类场景需谨慎分析（见场景3、5）。

**示例说明：** 将闭源风格迁移算法编译进 RapidRAW 并分发，与软件动态链接，它与 RapidRAW 共享内存和函数，此为**结合程序**，闭源模块也必须 GPL/AGPL 开源【76†L1258-L1262】【80†L161-L168】。若仅以独立服务形式部署（快速行程转换为远程 API），则更接近**分离程序**；但如果用户脱离该服务无法完成工作，许可证条款可能主张这仍是衍生作品【80†L165-L172】。总之，判断时应参考 FSF FAQ 中“单一结合程序”准则【76†L1258-L1262】，并权衡具体调用方式。

## 集成场景合规性分析  

以下情景综合了最常见的集成方式及其合规要求（以 AGPL-3.0 为准）。表格对比了每种情况是否触发开源义务和风险，并给出合规建议。

| 场景                         | 是否触发 AGPL 要求 | 法律风险              | 合规措施建议                                     |
|----------------------------|---------------|--------------------|---------------------------------------------|
| **1) 编译进二进制并分发**<br/>（闭源模块编译入 RapidRAW，作为单一程序分发） | 是            | 高：构成衍生作品，闭源违反 AGPL | 必须将整个软件（含闭源部分）按 AGPL 开源【76†L1258-L1262】【82†L123-L131】；或移除闭源模块。 |
| **2) 插件随安装包分发**<br/>（闭源模块作为独立进程/插件文件，与 RapidRAW 一起打包） | 通常视为是    | 中高：可能被认定为聚合或衍生，含糊不清 | 最安全做法：若插件与主程序紧密配合，视作衍生，需开源；若明确分离（如可自选安装路径、通过 IPC 标准协议调用），可尝试仅发布链接说明，但风险较大，应征询法律意见。 |
| **3) 后端云服务调用**<br/>（RapidRAW 本地调用远端闭源风格 API，不分发闭源代码） | 取决于*       | 低：本地不含闭源模块，网络调用规避直接分发 | RapidRAW 本身仍 AGPL，需对其修改开源；闭源风格服务独立托管不在本地分发，可保持闭源。但需确保接口非紧密链接，可视为分离【76†L1258-L1262】。提供网络服务时，按照 AGPL 要求向用户提供 RapidRAW 源码【61†L355-L363】。 |
| **4) 仅公司内部使用**<br/>（不向外分发，仅内部部署） | 否            | 极低：不构成“传输”  | AGPL 不要求公开源代码。建议仍保留 LICENSE 等合规文档，内部遵循许可义务。 |
| **5) 提供托管/SaaS 服务**<br/>（用户通过网页/云使用风格迁移功能） | 是            | 高：远程用户使用视为网络分发 | AGPL 要求“为网络用户提供源码”【61†L355-L363】【84†L108-L113】。可行策略：将 RapidRAW（或其核心）作为 SaaS 服务端运行，必须公开源代码；或仅提供核心服务而将风格算法作为独立服务架构。务必在界面上显著提供源码获取（如“获取源码”链接）。 |

*注：场景3 的合规性取决于后端服务的调用方式，如果 RapidRAW 中仅配置远程接口且风格算法未集成在可分发的二进制里，通常 AGPL 不要求公布闭源服务端代码。然而，RapidRAW 的修改版仍需开源【82†L123-L131】。场景3 若将调用逻辑视为轻量配置、与核心软件松耦合，则合规风险最低。

**说明：** 场景1和2中，若闭源模块功能与 AGPL 软件结合紧密，几乎肯定会被判定为衍生作品，必须开源【76†L1258-L1262】【80†L161-L168】。将闭源插件作为**可选扩展**并采用严格隔离（如独立进程通信、文档清晰声明）可减轻风险，但仍存在解释空间。场景3/5 (SaaS) 下，只要用户能够“远程使用” AGPL 软件，AGPL 第13条要求提供源码【61†L355-L363】【84†L108-L113】。如果闭源模块完全位于服务端且不随产品一起分发，可不公开该模块代码；但必须公开用于服务的 AGPL 代码。场景4（内部使用）最安全：AGPL 不管私有使用。 

## 许可证兼容性与风险  
将 AGPL-3.0 与其他开源许可混合使用时需注意兼容性：  
- **MIT/Apache-2.0**：均为宽松许可，兼容 GPL 家族，可与 AGPL 代码一起分发【84†L148-L152】。您可以在 AGPL 项目中包含 MIT/Apache 许可的组件，但需在分发时附上它们的原许可和版权声明。  
- **CC-RAIL（Stable Diffusion 等）**：稳定扩散模型采用的 CreativeML Open RAIL 许可附加了使用限制（如不得进行非法、亵渎等用途，要求衍生模型继承相同限制【85†L431-L439】）。这种许可**不属于自由/开源软件范畴**，与 AGPL 存在矛盾：GPL/AGPL 不允许增加额外使用限制【84†L148-L152】。若将 CC-RAIL 模型权重与 AGPL 项目一起分发，将可能违反 AGPL（因附加了额外限制）。应谨慎处理：最好只提供模型下载链接、不直接包含在发行包，或获得权利人额外授权。  
- **商业闭源模块**：AGPL 不兼容任何闭源软件。如果将商业代码以任何形式与 AGPL 代码结合分发或在同一产品中发布，则可能强制要求整体开源（除非明确按照“聚合作品”放置）【76†L1258-L1262】【80†L161-L168】。  
- **双重许可模型**：如果闭源模块也采用 GPL 兼容许可，可考虑双许可证模式（如双重 GPL/商业许可）。但必须注意：如果闭源模块授权为 GPL-兼容，结合仍需完全开源；若使用不同许可条款并仍随 AGPL 代码发布，则冲突。  

总之，除 MIT/Apache 外的许可证需逐一审查兼容性。特别是模型权重等第三方内容，应在分发时明确许可信息并遵守相应限制，以免侵犯版权或违反许可证。对于 CC-RAIL 等非商业许可的模型，商业分发前应咨询法律意见。

## 商业化策略建议  

下面列出数种常见商业化策略及其法律风险和实施难度评估：  

| 策略                     | 说明                                             | 法律风险             | 业务影响与难度                             |
|------------------------|------------------------------------------------|------------------|----------------------------------------|
| **1. 完全开源+付费支持/培训**    | 将 RapidRAW（包括风格迁移功能）作为 AGPL 开源项目发布，通过提供培训、服务协议等收费。 | 风险最低：符合 AGPL 要求【82†L129-L134】 | 实施难度低；可扩大社区；盈利依赖增值服务。可能吸引 OSS 用户，但收益模式依赖支持服务。 |
| **2. 双许可证策略**       | RapidRAW 主体继续开源，闭源风格模块另行获得权利人授权，可在商业许可下发行。 | 中等：需获原作者许可，否则违反 AGPL【76†L1258-L1262】 | 若可谈妥商业授权，可闭源销售给特定客户；成本高，需与原权利人协商许可。 |
| **3. 云端/API 服务模式**   | 将风格迁移功能仅作为云端服务提供，RapidRAW 本地执行调用，或将整体应用置于云端 SaaS。 | 风险较高：AGPL 要求为用户提供源码【61†L355-L363】 | 如果只提供 API，RapidRAW 修改版（AGPL）仍需开源；可不分发闭源模块。用户体验好，但需公开核心代码（或简化用户仅用文档指向源）。实施需建服务器成本。 |
| **4. “聚合”模式插件**      | 将闭源风格迁移作为独立可选插件/服务，明确定义插件API为“系统接口”，模仿Linux做法，主程式显式豁免。 | 风险高：GPL默认不允许闭源插件【76†L1258-L1262】；特殊许可例外需作者授权。 | 理论上可通过增加许可例外来允许闭源插件；需原作者自行向所有贡献者声明豁免（类似GPL特殊例外）【76†L1341-L1349】。实施复杂，需要项目版权持有人制定例外条款。 |
| **5. 替代开源方案**       | 使用兼容许可的开源风格迁移算法（如WCT/AdaIN等），完全替换闭源模块功能。 | 风险低：全部为开源替代，实现AGPL要求 | 可避免闭源问题，完全开源，但可能功能或质量不及原模块。开发工作量较大。 |
| **6. 仅内部部署/限定市场** | 仅针对内部或定制化客户使用，不在市场公开分发。 | 风险低：无分发，AGPL不触发【86†L1760-L1766】 | 无需开源，但无法通过广泛销售获利。适合少量企业客户；长期维护成本可能高。 |

每种策略都有**权衡**。完全开源方案法律风险最低，但需其他方式盈利；双许可或商业授权可闭源但复杂难行；纯云服务需同时满足用户获取源代码义务；定义“可例外接口”需项目权利人统一授权。推荐同时准备**法律咨询**，明确内部管理制度（如贡献者许可协议/版权声明），并严格记录使用的第三方组件及其许可，以支持所选策略。

## 文档与合规清单  

在产品发布时应准备并随发行包提供下列文件和说明：  
- **LICENSE 文件**：包含 AGPL-3.0 全文（或引用 FSF 官网），并在 README 中明确版权和许可证【89†L19-L22】【84†L108-L113】。  
- **NOTICE 或版权声明**：保留原 RapidRAW 版权声明，注明任何修改，并附加项目版权人信息。  
- **第三方许可证清单**：列出所有集成的开源组件（如 rawler、LensFun、ONNX 等）及其许可证，确保遵守各自要求。  
- **源码获取信息**：明示用户如何获取完整源代码（例如提供下载链接或源代码包），满足 AGPL “提供源码”要求。若以二进制分发，还需按照许可证第6条提供源码（通过附带或链接）【82†L123-L131】。  
- **模型/权重许可说明**：风格迁移用到的预训练模型（如 Stable Diffusion 权重）应附上相关许可协议或链接（如 CreativeML RAIL 许可【85†L431-L439】），并在分发协议中标注任何使用限制。  
- **用户协议与隐私政策**：若提供云服务（SaaS），需有用户协议说明服务条款，并提供隐私政策，明确数据使用和开源要求（AGPL 本身不管数据，但透明性和合规很重要）。  
- **分发包源代码**：根据 AGPL 第6节要求，若以物理介质或网络分发程序，应同时提供或书面承诺提供源代码。可以在文档中附上“获取源码方式”的说明或许可证链接。  
- **合规流程记录**：建议建立软件物料清单（SBOM），记录所用开源/第三方组件版本和许可证，方便内部合规审核。  

这些文档确保您满足 AGPL-3.0 的各种义务（如包含许可证、公布修改日志等）【84†L108-L113】【82†L129-L134】。若选择闭源插件或云服务模式，还应记录分隔措施（如接口规范）和任何与版权方的授权协议，以备审计。

## 结论与行动建议  

综上所述，**将闭源风格迁移直接编入 RapidRAW 并发布则必须开源**。AGPL-3.0 的强制性“传输即开源”条款适用于任何与 AGPL 程序结合的分发形式【82†L123-L131】【61†L355-L363】。如果希望保持闭源，可以考虑将风格功能作为外部服务提供，确保在本地分发时不包含闭源代码；但需注意 AGPL 对网络服务用户提供源码的要求【61†L355-L363】。对于仅限内部或专用部署，则无需向外部公布源代码【86†L1760-L1766】。

**推荐下一步：**  
- **咨询专业律师**：对具体集成方式进行法律审查，确认是否侵犯AGPL条款。  
- **清晰划分模块边界**：如果选择插件或服务模式，设计明确的接口（如进程、REST API），并在贡献者许可协议中说明这些接口为“独立模块”以降低衍生风险。  
- **准备合规文档**：创建并随发行包提供 License、NOTICE 和第三方许可证清单，维护 SBOM，确保满足 AGPL 要求。  
- **评估商业模式**：确定采用何种商业化策略（如开源商业支持或云服务），并据此调整架构与许可声明。  
- **备选方案研发**：考虑使用完全开源的风格迁移替代品或开发自家实现，以便可完全在 AGPL 体系内运营。  

本指南仅提供合规参考，具体实施应综合法律和业务需求。若需要法律意见或合同建议，请务必咨询专业律师。以上分析基于 AGPL 官方文本及 FSF/OSI 指南【82†L123-L131】【76†L1258-L1262】【84†L108-L113】【86†L1760-L1766】整理而成。
